message; /v1/auth/verify consumes the nonce, recovers the signature, atomically
resolves-or-mints the user, issues a session JWT + encrypted-cookie + CSRF, and
emits the full audit chain.

This is the auth-highest-scrutiny task. Every decision below is documented in
line because the operator was clear that the review gate would line-by-line.

## LBR-1 — TRANSACTIONAL resolve-or-mint

**Decision: pattern `withTransaction<T>(fn: (spine: SpinePort) => Promise<T>)` on `SpinePort`.**

The hazard: T1.5's `resolveOrMintByWallet` is non-atomic. Two concurrent
`/v1/auth/verify` calls for the same fresh wallet can BOTH:

1. `resolveByWallet` → null (no prior link)
2. `mintUser` → fresh user_id (each gets a different one)
3. `linkWallet` → ONE succeeds, the OTHER raises
   `uq_wallet_links_active_address` uniqueness violation

leaving the loser's user_id **ORPHANED** in `users`. Per T1.5 build notes
§Caveats: *"v1 accepts this tradeoff because the auth flow isn't yet wired
(T1.6) — the caller pattern that will need true atomicity lands when T1.6
builds the verify orchestrator."*

T1.6 lands the atomicity. Two layers:

### Layer 1 — `SpinePort.withTransaction`

Adds to the port (`packages/ports/src/spine.port.ts`):

```typescript
withTransaction<T>(fn: (spine: SpinePort) => Promise<T>): Promise<T>
```

Adapter impl (`packages/adapters/src/postgres-spine-adapter.ts:208-223`) wraps
Bun.SQL's native `sql.begin(async (txSql) => {...})`. The closure gets a
fresh `PostgresSpineAdapter` constructed with the txn-bound sql handle, so
every method call routes through the single reserved connection.

Native Bun.SQL semantics: `begin()` resolves to the closure's return on
COMMIT, re-throws on ROLLBACK. The inner adapter MUST NOT be retained past
the closure (its handle is invalidated).

### Layer 2 — `WalletLinkRaceError` + caller-side retry

The engine's `resolveOrMintByWallet` catches PG `23505` on the
`uq_wallet_links_active_address` constraint and re-throws as
`WalletLinkRaceError`. The caller (route handler) catches this, lets the
txn ROLLBACK, and re-runs `resolveOrMintByWallet` inside a fresh
`withTransaction`. The second pass's `resolveByWallet` hits the winner's
existing row and short-circuits — same user_id returned to BOTH callers.

The retry is capped at 1 attempt; a stable system converges. Two races in
a row on the same input is a different signal (genuine bug) and we let
the 500 surface.

### Why not `SpineConflictError(kind: 'wallet_link')`?

T1.5's `SpineConflictError` is raised from `linkAccount` + `claimNym` for
the D9 cross-user-collision policy. Reusing the same class for the
linkWallet race would conflate "this is a transient race, retry"
semantics with "this is a hard conflict, return 409" semantics.

`WalletLinkRaceError` is structurally different — it's NOT a cross-user
collision (no D9 applies). Separating gives the route handler a clean
typed signal for "retry, don't 409."

### Proof — the LBR-1 concurrent-race tests

```typescript
// src/api/__tests__/auth-flow.test.ts:466-540
it("LBR-1: TWO concurrent /verify calls for SAME wallet → both 200 with
    SAME user_id, ONE user row, ONE wallet_link", async () => {
  const pk = generatePrivateKey()
  const acct = privateKeyToAccount(pk)
  const wallet = acct.address.toLowerCase()

  // Two parallel challenges (distinct nonces)
  const [ch1, ch2] = await Promise.all([
    fetch(...challenge for wallet...),
    fetch(...challenge for wallet...),
  ])
  // Two parallel signs
  const [s1, s2] = await Promise.all([
    acct.signMessage({ message: c1.message }),
    acct.signMessage({ message: c2.message }),
  ])

  // THE RACE — both verify calls dispatched simultaneously
  const [v1, v2] = await Promise.all([
    fetch(...verify c1...),
    fetch(...verify c2...),
  ])

  // LOAD-BEARING ASSERTIONS
  expect(v1.status).toBe(200)
  expect(v2.status).toBe(200)
  expect((await v1.json()).user_id).toBe((await v2.json()).user_id)

  const users = await sql`SELECT user_id FROM users`
  expect(users).toHaveLength(1)  // ← ONE user

  const links = await sql`SELECT * FROM wallet_links WHERE unlinked_at IS NULL`
  expect(links).toHaveLength(1)  // ← ONE wallet_link
})
```

Plus a 10-way scale variant. Both pass on postgres:18.1 + Bun.SQL 1.3.x.

### Bun.SQL error-shape detail (commit a552e60 fix)

While running the E2E concurrent-race tests, the first impl 500'd on the
PG conflict — the error wasn't being classified as `WalletLinkRaceError`.

Empirical probe revealed Bun.SQL surfaces PG errors as `PostgresError` with:
- `code: 'ERR_POSTGRES_SERVER_ERROR'` ← driver tag, NOT SQLSTATE
- `errno: '23505'` ← the actual SQLSTATE
- `constraint: 'uq_wallet_links_active_address'` ← NOT `constraint_name`

My initial `isWalletLinkConflict` checked `e.code === '23505'` and
`e.constraint_name`. Fixed to check BOTH `.errno` AND `.code` for SQLSTATE,
and BOTH `.constraint` AND `.constraint_name` for the index name. Same
defense added to adapter's `isUniqueViolation`. Documented inline so the
next debug session starts from empirical truth, not guess.

## LBR-2 — SIWE message-builder closure

**Decision: pattern (B) — `messageBuilder: (nonce) => string` closure on
`MintNonceInput`.**

The hazard: SIWE messages (EIP-4361) embed the nonce verbatim:
```
identity-api wants you to sign in with your Ethereum account:
0x29bd32cc92cd9ddbacbb3c2b5cd30e44582a0483

Sign in to identity-api boot proof.

URI: https://identity-api.live-proof
Version: 1
Chain ID: 1
Nonce: osW9d3X8GvZakHvAg3ugWkFMK12symnHtlZbCAFEwkc    ← nonce HERE
Issued At: 2026-05-25T04:51:47.396Z
```

The adapter generates the nonce. Pattern (A) (two-write) would mint with
a placeholder message, build the real message, then `UPDATE auth_nonces
SET message = ... WHERE nonce = ...`. That:
- Costs 2 DB writes per challenge
- Leaves a window (between INSERT and UPDATE) where the row's `message`
  and `nonce` are INCONSISTENT — a forensic hazard if any read inspects
  the row in that window

Pattern (B) extends `MintNonceInput` to accept EITHER `message: string`
(T1.4 EIP-191 shape, preserved for back-compat) OR
`messageBuilder: (nonce: string) => string`. The adapter generates the
nonce, invokes the closure to construct the final message, INSERTs both
in ONE statement. No inconsistency window.

```typescript
// packages/ports/src/spine.port.ts:97-127
export interface MintNonceInput {
  readonly scheme: SpineNonceScheme
  readonly message?: string                          // T1.4 shape
  readonly messageBuilder?: (nonce: string) => string  // T1.6 LBR-2 shape
  readonly walletAddress?: string | null
  readonly ttlSec?: number
}
```

The adapter enforces "exactly one MUST be supplied" — supplying both is a
caller bug, supplying neither is a caller bug. Engine mirrors the guard
at its seam.

`MintNonceResult` also now carries the resolved `message` string — the
route handler doesn't have to re-run its own builder to return the
canonical signed string to the caller.

EIP-191 path: the closure trivially returns
`"identity-api login challenge: " + nonce`. Sietch precedent shape.

## LBR-3 — L7 patch (malformed JWT → 401, not 500)

**Decision: pattern (I) — wrap at `src/auth.ts`, keep vendored Hyper pristine.**

The hazard (verdict-L7): `src/hyper/auth-jwt/jwt.ts:verifyJwt` calls
`JSON.parse(b64urlToUtf8(headerSegment))` and the same for payload. A
3-segment token with garbage segments base64-decodes to non-JSON →
`SyntaxError` → leaks past the middleware's narrow
`catch (e) { if (e instanceof JwtError) }` → HTTP 500 on attacker input.

Auth surface must never 500 on caller input.

`hardenAuthMiddleware` in `src/auth.ts:107-145` wraps the installed auth
middleware: catches `SyntaxError` and `TypeError` → 401 'malformed_token'.
Lets genuine downstream errors propagate as 5xx.

Verified by trace probe + 3 regression tests in `routes.test.ts`:
- `total-garbage-not-a-jwt` (1 segment) → JwtError path → 401 'invalid_token'
- `aaa.bbb.ccc` (3 segs, base64-decodable but not JSON) → SyntaxError caught
  by L7 wrap → 401 'malformed_token' (was 500 baseline)
- `aGVsbG8.d29ybGQ.YWFh` (3 segs, decodes to "hello" / "world") → same
  SyntaxError path → 401 'malformed_token' (was 500 baseline)

Why (I) not (II) (patch vendored source):
- Keeps `hyper.lock.json` pristine (no drift to manage)
- Sprint-1.1 #5/#6 will land the upstream patch + remove this wrap
- The wrap is small (~30 lines) and grep-discoverable

## Signature verifier — viem (NOT Sietch-adapt)

**Decision: `bun add viem` + new `packages/adapters/src/wallet-signature.ts`
backed by viem's `recoverMessageAddress`.**

Trade-off:
- viem is a new direct dep (~50KB gzip)
- Sietch's existing `SignatureVerifier` already uses viem for the same
  primitive — we don't import Sietch's class because it's not exported as
  a workspace package + we want a function (not class) shape

Why NOT hand-roll secp256k1:
- Would mean owning ecrecover + Keccak digest + EIP-191 prefix +
  signature-format normalization
- viem is well-audited, used by honey-road + Sietch + score-api
- The vendored-source no-npm discipline applies to EXTERNAL consumers
  of identity-api (per PRD §4.3 "consumed as source-distributed"), NOT
  to internal infrastructure deps

Why viem's `recoverMessageAddress` works for BOTH SIWE and EIP-191:
- SIWE messages, despite their EIP-4361 structure, are SIGNED via
  `personal_sign` (EIP-191 envelope) — the recovery primitive is the SAME
- The scheme parameter is anchored at the call site for audit-row
  precision + future ERC-1271 dispatch (Sprint-1.x)

Discipline:
- Recover FIRST, then case-insensitive compare. Prevents the
  expected-equals-expected bug class.
- viem throws on bad sigs → CATCH → `ok: false, reason: 'recover_error'`.
  Auth surface never 500s on attacker input.
- Strict format pre-check (0x + 132 chars + hex-only) — Sietch's exact
  rule, single source of truth.
- 18 unit tests with ground-truth signatures from
  `viem/accounts.generatePrivateKey + .signMessage` (no fixed mainnet
  vectors needed).

NOT in v1: ERC-1271 (contract wallets / account abstraction). A
contract-wallet user attempting SIWE will fail recovery. Sprint-1.x
follow-up.

## JWT shape minted

`src/jwt-mint.ts:88-119` constructs the minimum-viable JWTClaim subset
matching `packages/protocol/src/jwt-claims.schema.json`:

```json
{
  "sub": "8df099b0-a1d7-4f4e-aefd-c8992796fab0",
  "wallets": [{ "chain": "ethereum", "address": "0x29bd...0483" }],
  "tenant": "freeside",
  "iss": "identity-api",
  "aud": "freeside",
  "iat": 1779684707,
  "exp": 1779688307,
  "jti": "996842f2-a59c-44bc-8715-e1091efaa16b",
  "v": 1
}
```

(Above is the literal payload from the boot-proof curl trace — see §K
below.)

Decisions:
- `tenant` and `aud` default to 'freeside' (a generic substrate-wide slug).
  Per-world variants (e.g., `tenant: 'mibera'`, `aud: 'mibera'`) land at
  T2.x when world context is known at session-mint time.
- `tier` / `display_name` / `discord_id` / `nft_id` left UNSET. These are
  populated by the compose edge (T2.x), not by the auth flow.
- `jti` is `crypto.randomUUID()` (UUIDv4). V1 NO denylist (per protocol
  schema docstring: "jti denylist is V2"). Unique identifier, not a
  revocation handle.
- `exp` is +1h from `iat` per PRD §4.4 FR-J2 (Lock-8).
- HS256 signing. Same JWT_SECRET as the verifier on `/v1/me`. Sprint-1.1
  #3 swaps to ES256 via `jose` + the `JWTSigner` port + own JWKS.

We DON'T import the protocol's `JWTClaimSchema.parse` for runtime
validation in v1 — keeping the construction inline + readable while the
shape stabilizes. When `LocalEs256Signer` lands (Sprint-1.1 #3),
construction will gate through `JWTClaimSchema.parse(payload)` before
signing.

## Cookie + CSRF integration

`/v1/auth/verify` is `applyWithSession(route.post(...))` — installs the
`session({...})` + `csrfGuard()` middleware pair (the L5 fix from T1.0).

Handler populates `ctx.session`:
```typescript
ctx.session.regenerate()              // fixation defense — rotate id on auth
ctx.session.set("user", { sub: userId, wallet })
ctx.session.set("jwt", session.token)
```

Hyper's session middleware writes `Set-Cookie: idapi_sess=...; HttpOnly;
SameSite=Lax; Secure` on response. csrfGuard lazily issues the
`csrf=...` cookie (non-HttpOnly so JS can echo it) on the first
established-session response.

Verified in boot proof §K: both cookies set on `/v1/auth/verify` response.

## Audit events emitted

End-to-end audit chain for a happy verify:

```
nonce_minted     (user=null, actor=self)     ← T1.4 engine
nonce_consumed   (user=null, actor=self)     ← T1.4 engine
user_minted      (user=<new>, actor=self)    ← T1.5 engine [iff first-time wallet]
wallet_linked    (user=<new>, actor=self)    ← T1.5 engine [iff first-time wallet]
auth_verified    (user=<known>, actor=self)  ← T1.6 route emits
```

For rejections:
```
nonce_rejected   (user=null, actor=self, payload.reason ∈ {unknown, used,
                  expired, scheme_mismatch})
auth_signature_rejected (user=null, actor=self, payload.reason ∈
                  {wallet_mismatch, malformed_signature, signature_mismatch,
                  recover_error})
```

Verified by `audit chain coverage` test in `auth-flow.test.ts` — asserts
the 5 happy events land in DB after a complete verify cycle.

## Caveats / deferred

- **ES256 swap → Sprint-1.1 #3.** Currently HS256 against `JWT_SECRET`.
  PRD §4.4 FR-J2 specifies ES256 + own JWKS at `/.well-known/jwks.json` +
  overlap-window key rotation. When swapped, `src/jwt-mint.ts` becomes
  the `mint-jwt-orchestrator` adapter calling the `JWTSigner` port; the
  HS256 inline signing goes away.

- **`JWTSigner` port + LocalEs256Signer adapter NOT YET wired.** PRD §4.4
  FR-J1's "construct claims, never sign" decree — currently we DO sign
  inline (HS256). The migration to the port pattern lands with the ES256
  swap (Sprint-1.1 #3).

- **`tenant: 'freeside'` is a generic default.** Per-world session minting
  (e.g., `tenant: 'mibera'`) lands at T2.x compose-edge when world
  context becomes known. The protocol's `assertTenantBoundary` helper
  consumes this; consumers MUST call it.

- **`audience: 'freeside'` is a generic default.** Per-world audience
  binding for I6 enforcement is T2.x's job.

- **`tier` / `display_name` / `discord_id` claims NOT populated.** The
  compose edge enriches the JWT (T2.x) with world-specific claims; T1.6's
  session mint is the minimum that unlocks /v1/me.

- **Profile claims → T2.x.** /v1/profile?world=:slug is FR-P1 / T2.x; this
  task only minted the session, not the read-time compose.

- **ERC-1271 (contract wallets) NOT supported.** EOA personal_sign only.
  Sprint-1.x follow-up; viem ships `verifyHash` for the 1271 dispatch
  pattern.

- **`jti` denylist V2.** Per protocol schema docstring: "jti denylist is
  V2 per SDD §11.5 D-H." Today jti is unique-identifier-not-revocation-
  handle. Logout + token invalidation will need the denylist (V2).

- **No /v1/auth/logout endpoint.** T1.6 mints sessions; tearing down a
  session is not part of T1.6 scope. The encrypted-cookie session can be
  destroyed by `ctx.session.destroy()` from a future logout route; JWT
  invalidation needs the V2 jti denylist.

- **`credential-bridge-siwe` + `credential-bridge-dynamic` NOT
  implemented.** Per T1.6 brief: T1.7 (arrakis-1ma8). Today the route
  calls `verifySignature` directly inline; T1.7 will refactor into the
  bridge abstraction.

- **L7 wrap is temporary.** When Sprint-1.1 #5/#6 upstream-patches
  `src/hyper/auth-jwt/jwt.ts` to catch SyntaxError inside `verifyJwt`,
  the `hardenAuthMiddleware` wrap becomes redundant + can be removed.
