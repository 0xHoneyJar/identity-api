## Design choices

### 1. Adapter: NEW `postgres-spine-adapter.ts`, NOT extending the cycle-B `postgres-split-adapter.ts`

Decision made up front + carried through. Rationale:

- `PostgresSplitAdapter` is the cycle-B per-tenant READ adapter that
  implements the polymorphic `TenantAdapter` port over `midi_profiles`.
  Its surface is `resolveCredential` / `fetchClaims` / `ping`; it's
  parameterized by `TenantConfig`; it uses `pg.Pool`.
- `PostgresSpineAdapter` is the central SoR for ONE schema (the spine).
  It uses `Bun.SQL` (matches the migration runner — zero extra deps), and
  exposes spine-specific bidirectional operations (mintUser, linkWallet,
  linkAccount, claimNym, setPrimary, writeAuditEvent).
- Mixing them would (a) bloat the cycle-B per-tenant adapter with
  unrelated write semantics, and (b) tightly couple two evolution paths
  that have nothing to do with each other (per-tenant read shapes evolving
  with future tenants vs central SoR mutation contract evolving with FR-R6).

### 2. Resolver layout: NEW `packages/engine/src/resolve-spine.ts` (no `resolve-tier.ts` exists)

The kickoff said "extend resolve-tier.ts if present, or new resolve-spine.ts
— justify." PRD v3.0 §4.2 promises *"the existing 4-tier `resolve-tier`
keeps its structure; Tier-4 (direct wallet) becomes primary, Tier-1
(dynamic_user_id) becomes backfill"* — but **`resolve-tier.ts` does NOT
exist in the cycle-B foundation** (`packages/engine/src/` ships only
`mint-jwt-orchestrator.ts`, `tenant-config.ts`, `tenant-registry.ts`,
`tenants.yaml`).

So this file establishes the **surface where the future tier algorithm
composes**. Today it's wallet-first single-tier; tomorrow `resolveByWallet`
becomes the Tier-4 entry of a fallback chain (Tier-3 wallet_groups, Tier-2
additional_wallets, Tier-1 dynamic_user_id). The route layer doesn't
change — only the engine internals.

### 3. Port-discipline refactor mid-build (the dependency-arrow correction)

I initially wrote `resolve-spine.ts` to import the concrete
`PostgresSpineAdapter` class from `@freeside-auth/adapters`. That's a
**wrong-direction dependency** under SDD §12.4 (engine reasons through
ports; adapters → engine).

Corrected before commit by extracting `SpinePort` into
`packages/ports/src/spine.port.ts`. The engine now depends only on
`@freeside-auth/ports` (the interface); the adapter `implements SpinePort`.
This is the same pattern cycle-B used for `TenantAdapter` (port) ↔
`PostgresSplitAdapter` (impl).

Benefit: enabled the engine tests in `resolve-spine.test.ts` to use a
hand-rolled `MockSpine` (no DB needed) — 21 cases that verify the
ORCHESTRATION (which spine calls happen, in what order, with what args)
without ever touching PG.

### 4. Conflict policy implementation (D9)

The kickoff specified: *"`UNIQUE(provider, external_id)` HARD-FAILS on
duplicate. Per D9 + cycle-c FR-L3: this is the `cross_user_collision`
case — surface to the route handler as a typed error so the endpoint
returns 409 `cross_user_collision`."*

Implemented as:

- Adapter: `linkAccount` + `claimNym` wrap PG `23505` (unique_violation)
  errors and raise `SpineConflictError(kind: 'linked_account' | 'world_identity', context: {...})`.
  The error class carries the conflict kind so the route layer doesn't
  have to introspect PG error codes itself.
- Engine: `linkAccountWithAudit` + `claimNymWithAudit` emit a
  `conflict_rejected` audit event (`user_id=null` for pre-resolution
  conflicts) BEFORE re-raising. NFR-5 audit trail captures every
  attempted write, including rejected ones.
- Route layer mapping is **deferred to T4.1** (POST /v1/link/verified-wallet)
  because that's the endpoint where D9 actually fires. The two existing
  T1.5 write paths (mintUser, linkWallet on first auth) don't have a D9
  surface — they're owned by the T1.6 auth flow.

PG error detection (`isUniqueViolation` in the adapter) checks
`code === '23505'` first, then `errno === '23505'`, then a message
substring match. This last fallback is defensive against Bun.SQL versions
that surface the constraint name in `.message` but not on `.code` — without
it, a real duplicate would propagate as a generic 500 instead of a 409.

### 5. Auth integration via `.auth()` sugar

GET /v1/me uses the L4-fix `.auth()` sugar from `src/auth.ts`. The chain:

```typescript
const meBuilder = route.get("/v1/me").meta({...}) as unknown as { auth: () => typeof routeBuilderShim }
export const me = meBuilder.auth().handle(async (c) => {
  const sub = c.ctx.jwt?.sub ?? c.ctx.user?.sub
  // ... validate as UUID, dispatch getIdentity(sub), return 200 | 401 | 404
})
```

The `as unknown as` cast is the documented workaround for Hyper's missing
`.auth()` type (`src/auth.ts:9-21` describes this verdict-L4 pattern).
T1.5 didn't change `src/auth.ts` — only consumed the existing surface.

JWT `sub` claim is validated as a UUID v4. A bearer JWT with a non-UUID
sub returns 401 `invalid_sub` (not 200) — guards against the T1.6 auth
flow accidentally minting a JWT with a malformed subject.

### 6. Address normalization at the engine seam

EVM addresses are stored canonical-lowercase per SDD §3.2 ("store canonical
lowercase"). The active-unique index `uq_wallet_links_active_address`
matches case-sensitively, so a mixed-case write would create a duplicate
row for the same human wallet.

Normalization lives in `resolve-spine.ts:330-336` (the engine seam) —
not in the adapter, not in the route layer. Rationale: callers (T1.6
auth flow, T4.1 link-verified-wallet) shouldn't have to remember to
lowercase; one place to fix is cleaner than three. Non-0x addresses (a
future Solana base58) pass through unchanged — adding chain-specific
normalizers is a one-line extension.

### 7. Singleton SpinePort + test seam

`src/api/spine.ts` exports `getSpine()` (lazy-build from DATABASE_URL) +
`__setSpineForTest(spine)` + `__resetSpineForTest()`. Route handlers call
`getSpine()` INSIDE `.handle(...)` so module-evaluation of route files
does NOT trigger a DB connection at import time. The integration tests
exploit this: they install a `MockSpine` via `__setSpineForTest` BEFORE
the first handler runs.

Naming follows the cycle-B `__TEST_*` convention used in
`tenant-registry.ts` — signals "test seam, not production knob."

## Bun.SQL wire-shape gotchas (uncovered by DB tests)

Two adapter fixes that surfaced when the test suite ran against a real PG:

### chain_ids empty-array binding

`${[]}` binds as the literal empty string, which PG rejects with
`22P02: array value must start with "{"`. Fix: build the PG array literal
explicitly + cast at the parameter site:

```typescript
const chainIdsLiteral = `{${chainIds
  .map((id) => `"${id.replace(/\\/g, "\\\\").replace(/"/g, '\\"')}"`)
  .join(",")}}`
await sql`
  INSERT INTO wallet_links (..., chain_ids, ...)
  VALUES (..., ${chainIdsLiteral}::text[], ...)
`
```

Note: chain_ids values are escaped for `"` and `\` in case future inputs
include them. Today the values are EVM chain IDs (`'1'`, `'8453'`) so it's
defense-in-depth — but the escape lives at the wire boundary where it
belongs.

### JSONB binding

`${JSON.stringify(payload)}::jsonb` stores the SERIALIZED STRING as a
JSONB string-value (not an object). On read, `payload->>'key'` returns
null and `row.payload.key` is undefined. Fix: pass the JS object directly:

```typescript
await sql`
  INSERT INTO audit_events (event_type, user_id, actor, payload)
  VALUES (${event_type}, ${user_id}, ${actor}, ${input.payload})
`
```

Bun.SQL serializes JS objects to JSONB natively when the column type is
JSONB. Verified empirically against postgres:18.1.

Both are documented as comments in the adapter so the next contributor
doesn't re-trip them.

## Caveats / deferred

- **setPrimary route endpoint NOT wired.** SDD §5.3 lists
  `POST /v1/wallet/:userId/primary` (FR-R5) but the kickoff said *"if SDD
  doesn't have it explicitly, leave it as engine surface only — T1.6 or
  T4.1 can wire later."* SDD §5.3 DOES list it, but it would need a
  session check (caller can only set primary on their own user_id), and
  the session check requires T1.6 auth wiring to be meaningful. Left as
  engine surface (`setPrimaryWithAudit`); T1.6 or a follow-up wires the
  route.

- **Audit `actor` defaults to `"system"`.** T1.6 auth will populate it
  with the session subject ('self'); T4.1 will use 'sietch-redirect'.
  Today every audit row written by T1.5's caller-less paths (e.g., the
  curl-driven manual writes) gets `actor='system'`. Documented in the
  AuditActor enum docstring.

- **resolveOrMintByWallet is not transactionalized.** A concurrent race
  where two requests with the same fresh wallet both miss the resolve and
  both attempt to mint+link results in one orphaned user_id (the loser's
  mint succeeds but its linkWallet raises). The losing caller catches and
  re-resolves; the orphan never resolves. v1 accepts this tradeoff —
  T1.6's verify orchestrator will own the transaction when it builds the
  full auth flow.

- **getIdentity issues 4 PG queries instead of 1 mega-JOIN.** Justified
  in the adapter comment: a single LEFT-JOIN-with-array-agg duplicates
  rows by Cartesian product (wallets × accounts × worlds); the four
  small indexed lookups are < 100ms p95 trivially and read cleaner.
  Revisit if profile composition (T2.3) shows it as a hotspot.

- **No `world_identity` user-PK collision differentiation at the route
  layer.** Both classes of `claimNym` collision (world-UNIQUE nym taken;
  user-PK user already has a nym in this world) raise
  `SpineConflictError(kind='world_identity')`. The route can disambiguate
  via the error context if v1 needs different UX; today both render the
  same "nym already claimed in this world" message.

- **Provider enum tightened in routes.** T1.1 stub allowed
  `twitter|dynamic`. The spine DDL CHECK constraint only allows
  `discord|telegram|dynamic_user_id`. Corrected the route Zod to match
  the storage truth — a `twitter` value at the wire would fail the
  spine write anyway, so rejecting at 400 (route layer) is the
  earlier-better error.

## Cross-task contract for T1.6 / T1.7 / T4.1

T1.5 publishes the following surface for downstream tasks:

- **T1.6 (auth-verify)** consumes `resolveOrMintByWallet` from
  `@freeside-auth/engine` — atomic-feel resolve-or-create for the
  /v1/auth/verify path. It returns `{ userId, minted }` so T1.6 can
  branch on first-time vs returning. T1.6 should pass `actor: 'self'` to
  populate the audit log with session-subject context.

- **T1.7 (credential swap)** consumes the same engine surface for SIWE
  flows + a future `linkCredentialBridge` helper (not in T1.5 scope) for
  the EIP-191 bridge.

- **T4.1 (link-verified-wallet)** consumes the FR-R6 write methods
  (`linkWalletWithAudit`, `linkAccountWithAudit`) PLUS the D9 conflict
  policy at the ROUTE layer (the engine surfaces typed errors; the route
  maps them to HTTP 409 envelopes). T4.1 should pass
  `actor: 'sietch-redirect'`.

The engine + adapter surface is stable — T1.6/T1.7/T4.1 can pin against
the current API without expecting churn from T1.5.
