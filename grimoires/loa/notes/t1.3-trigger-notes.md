# T1.3 — primary-wallet integrity trigger

T1.3 ships the *convenience* half of the FR-R5 contract. The *hard guarantee*
half — the partial-unique `uq_wallet_links_one_primary_per_user` — already
landed in T1.2 because the SDD calls them a paired contract that ship together
in 0001 + 0002. T1.3 is therefore narrowly: the function, the trigger, the
behavior tests, the rollback.

## Scope (what landed)

- `packages/adapters/src/migrations/0002_primary_wallet_trigger.up.sql` —
  `sync_primary_wallet()` plpgsql function + `trg_sync_primary_wallet`
  AFTER trigger on `wallet_links` (INSERT OR UPDATE OF is_primary).
- `packages/adapters/src/migrations/0002_primary_wallet_trigger.down.sql` —
  drops trigger then function, both `IF EXISTS` for replay safety.
- `packages/adapters/src/__tests__/primary_wallet_trigger.test.ts` —
  10 focused trigger-behavior tests (installation surface, legitimate
  two-statement swap + updated_at advancement, soft-unlink isolation,
  multi-user isolation, self-reset no-op, single-INSERT-first path,
  hard-guarantee raise on INSERT, hard-guarantee raise on UPDATE,
  down removes trigger+function, up after down restores behavior).
- `packages/adapters/src/__tests__/migrate.test.ts` (T1.2 test file —
  minimally updated): broadened 4 round-trip assertions from "expect 0001
  only" to "expect 0001 in the applied list" so T1.2's tests survive the
  addition of 0002. The `down` test now loops until the migration ledger
  is empty (rather than asserting a single down empties it), preserving

## SDD discrepancy (FLAG TO OPERATOR — read first)

**The SDD-spec'd trigger does NOT deliver the SDD's stated guarantee.** This
is the most important finding from T1.3 and the only one that needs operator
attention before T1.5 (resolve write logic).

SDD §3.2 prose claims:
> "the trigger is the convenience that makes 'setting a new primary clears
> the prior' **atomic** and keeps `users.primary_wallet` mirrored"

The literal SQL in the SDD (which T1.3 ships verbatim — function name,
trigger name, body, AFTER timing) does NOT make single-statement primary swap
atomic. Empirically verified against postgres:18.1 (test runs documented
below + manual psql session in T1.3 build):

- `UPDATE wallet_links SET is_primary=TRUE WHERE wallet_address=B` when
  another row is primary for the same user **RAISES** `duplicate key value
  violates unique constraint "uq_wallet_links_one_primary_per_user"`.
- Cause: with `AFTER INSERT OR UPDATE OF is_primary`, PostgreSQL writes the
  new tuple version → partial-unique index check fires → conflict → ERROR.
  The AFTER trigger never gets to run its demote pass.

**Two possible reconciliations** (operator decides):

**(A) Amend SDD to BEFORE.** Empirically verified that switching the trigger
timing to `BEFORE` *does* deliver single-statement atomic swap (demote runs
first, then PG checks the partial-unique on the new tuple — no conflict).
Cost: SDD amendment, follow-up patch to 0002 (cheap — one line). Benefit:
T1.5 resolve logic can use the simpler one-statement promote pattern.

**(B) Keep AFTER; constrain caller workflow to two statements.** T1.5
resolve logic MUST use the two-statement idiom (demote prior, then promote
new — same transaction). The trigger then handles only the mirror to
`users.primary_wallet`. Cost: a hard contract for T1.5 callers; a more
fragile invariant (two-statement order is implicit, not enforced). Benefit:
SDD ships unchanged.

**T1.3 ships interpretation (A's intermediate state)**: the SDD-prescribed
shape verbatim (AFTER), with tests asserting the actual two-statement
delivery, and this build note flagging the choice back to the operator.
T1.5 build should resolve this before authoring write logic. Until then,
T1.5 should follow interpretation (B) (two-statement workflow) defensively.

The test suite documents BOTH behaviors:
- Test 3 (`legitimate two-statement swap`) exercises the workflow that
  works under AFTER.
- Tests 7-8 (`hard guarantee INSERT` + `hard guarantee UPDATE ALSO raises`)
  prove the partial-unique fires before the AFTER trigger on both naive
  paths. These tests would FAIL (i.e., not raise) under interpretation (A)
  with a BEFORE trigger — they're the contract docs for the AFTER choice.

## Design choices (locked by SDD, documented here)

### AFTER vs BEFORE (shipped: AFTER)

The SDD prescribes **AFTER INSERT OR UPDATE OF is_primary**. T1.3 ships
this verbatim. See the SDD discrepancy section above for the empirical
finding that AFTER does NOT deliver single-statement atomic swap (the SDD's
prose claim). Operator must decide between (A) amend SDD to BEFORE and
(B) constrain caller workflow.

- The partial-unique is the **hard guarantee**: it makes two simultaneous
  primaries impossible at the storage layer. A caller who naively writes
  `INSERT ... is_primary=TRUE` for a user that already has a primary gets a
  uniqueness violation — by design, not by accident.
- The trigger is the **convenience for the legitimate workflow**: the
  caller demotes all prior primaries in one UPDATE, then promotes the new
  one in a second UPDATE (typically in one transaction). The trigger then
  fires on the second UPDATE, mirrors `users.primary_wallet`, and its own
  demote pass finds nothing to do (the caller already did it).

The test "hard guarantee" cases (tests 7 and 8) demonstrate the partial-unique
fires on both naive INSERT and naive UPDATE *before* the AFTER trigger gets a
turn — and assert the uniqueness-violation error message contains the index
name. That's the executable contract documentation.

### OF is_primary clause

The trigger's UPDATE arm has `OF is_primary` (the INSERT arm does not — PG
syntax disallows OF on INSERT). Effect: a pure `UPDATE wallet_links SET
verified_at = NOW() WHERE …` does NOT fire the trigger. This avoids write
amplification on every chain_ids / verified_at touch.

### IF guard inside the function

`IF (NEW.is_primary = TRUE AND NEW.unlinked_at IS NULL) THEN … END IF`:

- **`NEW.is_primary = TRUE`** — a row inserted/updated with `is_primary=FALSE`
  shouldn't do anything; the demote pass would do nothing useful and the
  mirror would corrupt `users.primary_wallet` with a non-primary address.
- **`NEW.unlinked_at IS NULL`** — soft-unlinked rows must not compete for
  primary; if the trigger fires on a soft-unlink UPDATE that also touches
  `is_primary` (operationally unusual but legal), we'd mirror an inactive
  wallet onto users. The guard prevents that.

### Self-exclusion in the demote pass

The trigger's demote query has `wallet_address <> NEW.wallet_address`. Without
it, the trigger would set the new primary's own row back to `is_primary=FALSE`
(an UPDATE collides with itself in PG's AFTER trigger context). Test 7
("re-setting an already-primary row to primary is a no-op") explicitly covers
the self-update path.

### Mirror updates `updated_at`

`UPDATE users SET primary_wallet = NEW.wallet_address, updated_at = NOW()` —
two fields. The `updated_at` advancement is intentional: downstream consumers
(score-api, world dashboards) may cache `users` rows and need a timestamp to
invalidate when the primary swaps. Test 2 (swap-on-UPDATE) explicitly asserts
`updated_at` advances past the prior mirror's value to guard the contract.

### Soft-unlink semantics

When a wallet is soft-unlinked (`unlinked_at` set), its `is_primary` field is
intentionally **preserved**: the historical record stands. The trigger refuses
to compete inactive rows for primary (the demote `WHERE` clause requires
`is_primary = TRUE AND unlinked_at IS NULL`), and the mirror also gates on
`unlinked_at IS NULL`. Test 5 covers this with a soft-unlinked A and a new
primary C — A's historical `is_primary=TRUE` is untouched; C wins.

## Edge cases considered, NOT deferred

- **Same-statement INSERT of two primaries for the same user** — impossible
  via the partial-unique. The first row's INSERT writes the partial-unique
  entry; the second row's INSERT raises before AFTER triggers fire.
- **UPDATE of an unlinked row to `is_primary=TRUE`** — the guard refuses to
  fire the demote-and-mirror logic; the partial-unique excludes inactive
  rows so no conflict arises. Net: a no-op for live state.
- **Re-setting the same primary to `is_primary=TRUE`** — self-exclusion in
  the demote pass + idempotent mirror. Test 7.
- **Multi-user concurrency** — `user_id` scope in the demote pass and the
  mirror. Test 6 demonstrates U1's change does not touch U2.

## Locking concerns

The trigger's UPDATEs take row-level locks on `wallet_links` (demote pass) and
`users` (mirror). Both are scoped by `user_id` so cross-user concurrency is
fine. Within a single user, the partial-unique serializes primary changes
(any two transactions attempting to set primary for the same user contend on
the same partial-unique slot — one wins, one retries). No explicit `FOR
UPDATE` is needed; partial-unique provides the serialization.

## Edge cases NOT covered (deferred to T1.5 resolve logic)

- **`primary_changed` audit event emission** — the SDD's audit_events table
  (`event_type IN ('wallet_linked'|'wallet_unlinked'|'primary_changed'|…)`)
  is a SEPARATE concern. A future T1.5+ commit will write to `audit_events`
  from the resolver, not from this trigger; logic in a trigger is hard to
  unit-test and risks tight-coupling DDL to event semantics.
- **De-duplicating spurious `is_primary=TRUE` UPDATEs that aren't actually
  changes** — the OF clause limits trigger firing to is_primary touches, but
  the function does not check `OLD.is_primary = NEW.is_primary` before doing
  work. For self-resets the demote pass is a no-op (self-exclusion) and the
  mirror is a one-row UPDATE that writes the same value — net cost is one
  cheap UPDATE per spurious call, judged acceptable for v1.

## Apply / rollback proof

When `TEST_DATABASE_URL` is set against a scratch DB, the test suite proves:

- Up → both 0001 + 0002 applied; `information_schema.triggers` returns 2 rows
  for `trg_sync_primary_wallet` (one per INSERT/UPDATE event_manipulation),
  both with `action_timing = 'AFTER'`.
- Swap-on-UPDATE → prior primary demoted (`is_primary=false`), new primary set,
  `users.primary_wallet` mirrored, `users.updated_at` advanced.
- Down → trigger and function gone; partial-unique survives (hard guarantee
  not affected by convenience-trigger rollback).
- Up after down → behavior round-trips cleanly.

Without `TEST_DATABASE_URL` the suite skips with a single info-line (same
posture as T1.2 — CI without PG passes; only ops with a real DB exercise it).

## Cross-task contract for T1.5

T1.5 (resolve / write-path logic, bead arrakis-232n) MUST use the
**two-statement swap pattern** until the operator resolves the SDD
discrepancy. Single-statement promote does NOT work with the SDD-prescribed
AFTER trigger (it raises uniqueness violation — empirically verified, see
test 8).

### Pattern A — wallet is already linked, just changing which is primary

```sql
BEGIN;
-- step 1: caller demotes any prior primary for this user
UPDATE wallet_links SET is_primary = FALSE
 WHERE user_id = :user_id
   AND is_primary = TRUE
   AND unlinked_at IS NULL;
-- step 2: caller promotes the new primary; trigger fires and mirrors
--          users.primary_wallet
UPDATE wallet_links SET is_primary = TRUE
 WHERE wallet_address = :new_addr AND user_id = :user_id;
COMMIT;
```

### Pattern B — adding a new wallet as primary on an existing user

```sql
BEGIN;
-- step 1: insert new row as non-primary
INSERT INTO wallet_links (wallet_address, user_id, is_primary)
VALUES (:new_addr, :user_id, FALSE);
-- step 2: caller demotes prior primary
UPDATE wallet_links SET is_primary = FALSE
 WHERE user_id = :user_id
   AND is_primary = TRUE
   AND unlinked_at IS NULL;
-- step 3: caller promotes the new row; trigger fires and mirrors
UPDATE wallet_links SET is_primary = TRUE
 WHERE wallet_address = :new_addr AND user_id = :user_id;
COMMIT;
```

### Pattern C — first ever wallet on a new user

```sql
INSERT INTO wallet_links (wallet_address, user_id, is_primary)
VALUES (:new_addr, :user_id, TRUE);
-- single statement; no prior primary to conflict; trigger mirrors users.primary_wallet
```

### What WILL raise (catch as contract-enforcement signal)

- `INSERT ... is_primary=TRUE` for a user that already has a primary
  → `duplicate key value violates unique constraint
     "uq_wallet_links_one_primary_per_user"`
- `UPDATE ... SET is_primary=TRUE` on a non-primary row while another
  primary exists for the same user → same uniqueness violation

### If operator chooses interpretation (A) — amend SDD to BEFORE

Then Pattern A simplifies to ONE statement (the demote step disappears —
the trigger does it). Pattern B simplifies to two statements (INSERT
non-primary + UPDATE to primary). Pattern C unchanged. The hard-guarantee
raises ALSO disappear; the trigger smooths them.
