## Field-by-field choices

### `schema_version: "3"`
Hard-required literal (`Schema.Literal("3")`).

### `slug: "identity-api"`
Matches the `^[a-z][a-z0-9-]*-api$` regex (ADR-008 ┬ºD-11 "*-api convention"). The
repo move (`freeside-identity` ÔåÆ `identity-api`) is captured in the PRD/SDD; the
slug is the canonical identity ahead of the GitHub rename (per PRD ┬º11 Q2).

### `publisher: "0xHoneyJar"`
Matches the inventory fixture publisher exactly. Per `project_freeside-npm-scope-and-consume`
memory: npm scope is `@0xhoneyjar` (NOT `@freeside`).

### `is.one_liner` (105 chars)
Exactly the SDD ┬º4.1 line. Below the 120-char cap.
> "Central identity SoR: resolve one human across wallets/accounts/worlds; serve composed per-world profiles"

### `is.scope` (3 entries)
SDD ┬º4.1 verbatim. The schema requires 2ÔÇô7; 3 is intentional ÔÇö the SoR is one
thing with three load-bearing surfaces (auth ┬À spine ┬À compose). More entries
would dilute the identity statement.

### `is_not` (4 entries) ÔÇö extends SDD's 3 by ONE
Bead requirement: ÔëÑ2. Each entry starts with `Does NOT|Will NOT|Refuses to` per
the `IsNotEntry` filter. PRD ┬º4.1 FR-B2 named **4** explicit anti-scope items:
*"does NOT index chains / compute score / store profile content / own the credential UI."*

SDD ┬º4.1 collapsed these into 3 entries by merging "index chains" + "compute score"
into a single line ("Does NOT index chains or compute score ÔÇö composes inventory-api / score-api / codex").

**I authored the union (4 entries)** because (a) the bead spec says "ship all of
them," (b) PRD authority > SDD authority on FR text, and (c) the schema cap is 7,
so 4 is comfortably within bounds. The 4 entries:

1. `Does NOT index chains or compute score ÔÇö composes inventory-api / score-api / codex` (merges 2 PRD items, same as SDD)
2. `Will NOT store per-world content (bios, dimensions, holdings) ÔÇö the spine is resolution-only`
3. `Refuses to call any third-party auth provider (Dynamic) in the auth path`
4. `Does NOT own the credential UI ÔÇö wallet-connect / passkey / SeedVault are pluggable client-side` *(added per PRD ┬º4.1 FR-B2; absent from SDD ┬º4.1)*

This is the only deliberate divergence from SDD ┬º4.1's literal yaml.

### `composes_with` (3 siblings)
Per PRD ┬º4.1 FR-B2: `{inventory-api, score-api, codex}`. Authored exactly as SDD ┬º4.1:

- `inventory-api` (required: false) ÔÇö holdings source
- `score-api` (required: false) ÔÇö score source
- `codex` (required: true ÔÇö SDD ┬º4.1 sets this; the Mibera 7-dim is the most-load-bearing read leg)

Note: `codex` is the existing slug (NOT `codex-api`). The `composes_with` key regex is
`^[a-z][a-z0-9-]*$` (lowercase-kebab, NO `-api` suffix required for composition keys ÔÇö
only `slug` enforces the suffix). Validated.

Tag hashes are PLACEHOLDERS (`InventoryPort@1.0.0+0000000000000000` etc.) satisfying
the `^[A-Z][A-Za-z0-9]*@\d+\.\d+\.\d+\+[a-f0-9]{8,16}$` regex. Per the inventory
fixture comment and PRD ┬º4.1: `freeside-cli doctor` (T2a) recomputes against real
port schemas at validation time. Do not hand-fabricate.

### `cycle_state`
- `status: candidate` ÔÇö Sprint 1 (Hyper runtime + spine stubs + DDL + trigger landed; reads/writes pending T1.5+). The other valid statuses (`active`, `mature`, `sunset`, `legacy`) all require production deployment; we're not there yet.
- `since: "2026-05-24"` ÔÇö today (ISO date).
- `next_review: "2026-08-24"` ÔÇö +92 days. Schema requires next_review Ôëñ +180d (doctor-enforced); 92d gives a reasonable Sprint-2/3 horizon.

### `mcp` (transport block ÔÇö OPTIONAL but present)
Per PRD ┬º4.1 FR-B5 the building deploys MCP under the federation gateway. The
mcp block uses BeaconV2's `McpBlock` shape:

- `shape: data` ÔÇö read-shaped (matches inventory-api ┬À resolve+profile are lookups)
- `paths: [remote-http]` (matches gateway-tenant pattern)
- `remote: streamable-http @ ${MCP_REMOTE_ENDPOINT}` (Railway-resolved at deploy)
- `auth: api-key` with header `X-MCP-Key` and credential `MCP_IDENTITY_UPSTREAM_KEY` (matches inventory's MCP_INVENTORY_UPSTREAM_KEY pattern; Auth's refine enforces both header AND credentials_ref when kind=api-key)
- `capabilities: [tools]` (single MCP capability ÔÇö tools/list)
- `tools` (3 entries ÔÇö read-tools only):
  - `resolveByWallet`
  - `getProfile`
  - `getMiberaDimensions`
- `pricing: free` (first-party tenant)
- `source_of_truth.type: database` (the spine Postgres)

**Why only 3 tools surfaced, not all 10 capabilities?** Per SDD ┬º4.1 verbatim.
The 7 non-exposed capabilities all require session state (auth flows, /me,
write-side linking) and are NOT MCP-tenant-callable from agents. The MCP surface
is for unauthenticated read-time agent composition (resolve + profile reads).
The other 7 are first-party-only via SDK/cookie/JWT. This is deliberate scoping,
not omission.
