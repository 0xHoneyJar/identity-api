## 1. Discovery findings (per port — vendored vs hand-authored)

For each of the 3 upstream buildings, the operator's runbook asked which
source-of-truth we built from. The local-checkout audit (`~/Documents/GitHub/{inventory,score,mibera}-api`)
turned up three different transport postures, and the decision tree below
captures what landed for each.

### inventory-api — hand-authored Zod (NOT vendored)

- **Local source:** `~/Documents/GitHub/inventory-api/` exists. Package is
  `@freeside/inventory` (`private: true` from `package.json`).
- **Transport posture:** **library-only today**. The package exposes
  `getHoldings(address)` as an in-process function (read fixtures or live
  GraphQL via `SONAR_GRAPHQL_ENDPOINT`). There is NO `bin/http.ts` and no
  Hono/Hyper service in the repo. The registry's
  `beacon_url: https://inventory.0xhoneyjar.xyz` is the post-HTTP-graduation
  target — graduation is real pending work.
- **Why we hand-authored:** vendoring the library would couple identity-api
  to the in-process function shape (and the npm package isn't published —
  `private: true`). Per FR-P3 no-embed, federation lives at the HTTP boundary
  even when both buildings ship from the same operator. The Zod schema in
  `packages/protocol/src/api/federation/inventory.ts` mirrors
  `HoldingsResponse` from `~/Documents/GitHub/inventory-api/types.ts:1-30`
  verbatim — the LIBRARY is the source-of-truth shape; the HTTP wrapper is
  the not-yet-existent transport.
- **Path used:** `GET /v1/holdings/:wallet` (projected REST shape — simplest
  HTTP wrapping of the library function). When the HTTP service ships, only
  the URL/path may move; the body schema is sealed.

### score-api — hand-authored Zod (NOT vendored)

- **Local source:** `~/Documents/GitHub/score-api/` exists. Package is
  `score-api` (`private: true`).
- **Transport posture:** **LIVE Hono service**. `src/routes/wallets.ts`
  line ~120 defines `GET /v1/wallets/:address` returning
  `walletService.getProfile(address)` — the `WalletProfile` interface at
  `src/types/api.types.ts:25-98`.
- **Why we hand-authored:** score-api is `private: true` — no npm
  distribution path. The package has its own type definitions (Drizzle ORM,
  Supabase clients) that would drag a massive dep cone into identity-api.
  The Zod schema in `packages/protocol/src/api/federation/score.ts` mirrors
  the public response shape verbatim and uses `.loose()` so forward-compat
  field additions on the score-api side don't reject the response — but
  field omission (missing field) DOES reject, catching backwards-incompat
  changes.
- **Path used:** `GET /v1/wallets/:address` (real endpoint, line ~120).
- **Auth:** `X-API-Key` per `src/middleware/auth.ts::authMiddleware`. The
  adapter takes the key via env (`SCORE_API_KEY`); absence yields a 401 →
  `unauthorized` at runtime, NOT a boot fail-fast.

### mibera-codex — hand-authored Zod against PROJECTED REST shape

- **Local source:** `~/Documents/GitHub/mibera-codex/` exists. Package is
  `@0xhoneyjar/construct-mibera-codex` (PUBLIC npm).
- **Transport posture:** **MCP-only**. `bin/http.ts` boots a Hono service
  but only serves `/healthz`, `/.well-known/mcp.json`,
  `/.well-known/beacon.json`, and the MCP JSON-RPC route at `/mcp`. The
  `lookup_mibera` surface is reachable ONLY via MCP tool-call — there is
  NO plain REST `GET /v1/mibera/:tokenId` route.
- **Why we hand-authored (and projected):** three options on the table.
  1. **Vendor the library + bundle the JSONL**: would pull ~10k lines of
     JSONL into identity-api's bundle; the data shape can drift without
     identity-api noticing. **Rejected**.
  2. **Wire an MCP client**: requires `@modelcontextprotocol/sdk`, session
     lifecycle, tool-call error semantics. **Rejected as out-of-scope for
     a port-foundation task** — that's T3+ infrastructure.
  3. **Hand-author Zod against a PROJECTED REST shape**: `GET /v1/mibera/:tokenId`
     and `POST /v1/mibera/batch` — the simplest HTTP wrappers around
     `lookupMibera`. The port is transport-agnostic; an `McpCodexAdapter`
     can satisfy `CodexPort` later if we ride MCP instead of waiting on
     codex to add the HTTP wrapper. **Chosen.**
- **Path used:** `POST /v1/mibera/batch` (`getMiberaTraits` batch shape —
  T3.1 will fetch the wallet's full Mibera holdings in one round-trip after
  `InventoryPort.getHoldings`).
- **Schema:** mirrors `MiberaEntry` from
  `~/Documents/GitHub/mibera-codex/src/types.ts:46-78` verbatim.
- **Auth:** none (codex beacon: `pricing: free`, `auth: none`).

## 2. Result-shape choice — discriminated union (NOT throws)

`FederationResult<TData> = { ok: true, data } | { ok: false, reason: FederationFailure }`

**Rationale (from the operator's runbook):** "Discriminated union is more
T2.2-friendly (no try/catch in the fan-out)." Confirmed during the build:

- T2.2's `Promise.allSettled` always wraps an inner throw into
  `{ status: 'rejected', reason: unknown }` — losing typed-error knowledge.
- With the discriminated-union contract, T2.2 writes a single
  `Promise.all([...])` (NOT `Promise.allSettled`) and every entry is a stable
  awaitable result; failures don't reject the promise. No try/catch, no
  instanceof, no allSettled boilerplate.
- The adapter resolves its promise no matter what — only programmer-error
  paths (Zod schema bugs, internal logic bugs) cause an actual rejection.
  Network / HTTP / parse / abort failures all show up as `ok: false`.

The 6 failure kinds (`timeout | unauthorized | not_found | upstream_5xx |
parse_error | network_error`) cover every wire-level outcome; T2.2 chooses
how to project each into the `degraded[]` array (the operator's prior
decision: each missed source is one string in `degraded[]`, structured
diagnostic context stays in the audit trail not the wire response).

## 3. URL / config strategy

- **Default URLs** per `packages/freeside-registry/registry.yaml`:
  - `INVENTORY_API_URL` → `https://inventory.0xhoneyjar.xyz`
  - `SCORE_API_URL`     → `https://score.0xhoneyjar.xyz`
  - `CODEX_API_URL`     → `https://codex.0xhoneyjar.xyz`
- **Singleton wiring** at `src/api/{inventory,score,codex}.ts` mirrors
  `src/api/spine.ts`'s lazy-build pattern.
- **CRITICAL difference from spine.ts**: federation singletons NEVER
  fail-fast. spine.ts throws on missing `DATABASE_URL` because the spine
  is load-bearing. Federation: missing baseUrl falls back to registry
  defaults; missing api key becomes 401 at compose time; per-source timeout
  + `degraded[]` absorbs both. /v1/profile stays 200 even when sources
  missed (FR-P3 graceful-degrade).
- **Test seams**: `__setInventoryForTest` / `__setScoreForTest` /
  `__setCodexForTest` mirror spine.ts's `__setSpineForTest` shape. Used
  by T2.2 + T2.3 tests + the federation-singletons.test.ts that lives in
  this commit.

## 4. AbortSignal handling — timeout vs network_error disambiguation

The shared `federationHttpCall` helper at
`packages/adapters/src/federation-http.ts` handles the load-bearing
AbortError disambiguation:

```ts
// Distinguish AbortError-due-to-signal (timeout) from AbortError-due-to-
// connection-failure (network_error). The AbortController's `aborted`
// flag is the source of truth at catch time.
const isAbort = isAbortError(err)
const callerAborted = opts.portOpts?.signal?.aborted === true
if (isAbort && callerAborted) {
  return failure({ kind: "timeout", ... })
}
return failure({ kind: "network_error", ... })
```

Why this matters: T2.2's per-source timeout (FR-P2) fires by calling
`AbortController.abort()` after the deadline; the resulting AbortError MUST
classify as `timeout` so T2.2's metrics + `degraded[]` aggregation can
distinguish "upstream too slow" from "upstream unreachable". A server-side
mid-stream abort produces the SAME AbortError name; the only signal is
`signal.aborted`.

Test coverage: `federation-http.test.ts` cases "AbortError + caller signal
aborted → timeout" and "AbortError + caller signal NOT aborted →
network_error (not timeout)".

## 6. T2.2 integration sketch

T2.2 (`compose fan-out orchestrator`, bead arrakis-l06n) will look like:

```ts
// packages/engine/src/compose-profile.ts (T2.2 sketch)
import type { SpinePort, InventoryPort, ScorePort, CodexPort } from "@freeside-auth/ports"

export async function composeProfile(opts: {
  spine: SpinePort
  inventory: InventoryPort
  score: ScorePort
  codex: CodexPort  // optional in v1 — T3.1 wires codex consumption
  userId: string
  primaryWallet: string
  perSourceTimeoutMs?: number  // default 2000ms per SDD §5.4
}): Promise<ProfileResp> {
  const identity = await opts.spine.getIdentity(opts.userId)
  if (!identity) throw new Error("user_not_found")

  // Per-source AbortController + setTimeout
  const ms = opts.perSourceTimeoutMs ?? 2000
  const invCtrl = withTimeout(ms)
  const scoreCtrl = withTimeout(ms)

  const [holdingsRes, scoreRes] = await Promise.all([
    opts.inventory.getHoldings({ walletAddress: opts.primaryWallet }, { signal: invCtrl.signal }),
    opts.score.getScore({ walletAddress: opts.primaryWallet }, { signal: scoreCtrl.signal }),
  ])
  invCtrl.clear()
  scoreCtrl.clear()

  const degraded: string[] = []
  if (!holdingsRes.ok) degraded.push(`inventory:${holdingsRes.reason.kind}`)
  if (!scoreRes.ok) degraded.push(`score:${scoreRes.reason.kind}`)

  return {
    identity,
    ...(holdingsRes.ok ? { holdings: holdingsRes.data } : {}),
    ...(scoreRes.ok   ? { score: scoreRes.data }       : {}),
    ...(degraded.length ? { degraded } : {}),
  }
}

function withTimeout(ms: number) {
  const ctrl = new AbortController()
  const timer = setTimeout(() => ctrl.abort(), ms)
  return { signal: ctrl.signal, clear: () => clearTimeout(timer) }
}
```

Key points:
- Both calls run as `Promise.all` (NOT `allSettled`); no failure rejects
  the outer promise because the discriminated-union contract resolves
  every adapter promise.
- `degraded[]` strings carry the failure kind tag; structured context lives
  in the audit log (per FR-P2 + the operator's prior decision about wire
  vs audit separation).
- T3.1 (Mibera dimensions resolver) plugs in after T2.2 lands — it'll
  consume `holdingsRes.data` to extract Mibera tokenIds and call
  `opts.codex.getMiberaTraits(...)` as a fourth fan-out leg.

## 7. Caveats

- **Projected REST shapes for inventory-api + codex**: documented in §1 +
  per-file provenance docstrings. When the upstream transports land, the
  Zod schemas and the URL config may need touch-up — bracketed by their
  respective sealed wire schemas (single source of truth).
- **score-api .loose() schema**: forward-compat tolerance is intentional;
  the trade-off is that score-api silently-added fields don't reach
  T2.2's compose layer until a follow-up commit tightens the schema. The
  alternative (strict) would reject every score-api PR that adds a field.
- **No CI runtime against real upstreams** in this commit. The mocks +
  the unit tests cover the adapter contracts; a full integration smoke
  against the production beacons happens at the cycle-c deployment gate.
- **`tsc --noEmit` from root surfaces 6 pre-existing errors in `src/hyper/`**
  — vendored tree, excluded from `tsconfig.json::include` but still
  surfaced when running from root. NOT introduced by T2.1.
