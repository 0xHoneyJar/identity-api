# T1.7 — credential swap build notes

T1.7 formalizes T1.6's inline `verifySignature` call into per-scheme
credential bridges, and locks the Dynamic SDK out of the live auth
path at TWO independent layers (Zod enum + runtime flag) PLUS a CI/dev
quarantine script that fails the build if `@dynamic-labs/*` lands in
any live-path source file.

## Decision log (the choices that mattered)

### D1 — Bridge interface shape (uniform across schemes, NOT per-scheme functions)

The first design fork: do bridges share a single interface, or does each
scheme get its own function shape?

Chose **uniform interface** (`CredentialBridge` with `verify(VerifyInput)`).

| Option | Pros | Cons |
|---|---|---|
| Uniform interface (chosen) | Single dispatch table in the route; uniform audit-emit; new schemes don't touch the router | Discriminated `VerifyInput` union; bridges need scheme-guard at runtime (defense-in-depth) |
| Per-scheme function shape | Each bridge's signature is precisely typed; no union | Route handler has scheme-specific conditionals; refactoring one scheme touches the route |

The uniform interface won on the **single-source-of-truth** axis: audit
rows for rejection reasons share a vocabulary (`VerifyRejectionReason`),
the route's dispatch is a 3-line lookup, and adding a new scheme is a
new module + a registry-table line (no router rewrite). The cost (a
discriminated union for the input + a runtime scheme-guard in each
bridge) is small and pays back at the type layer.

Defense-in-depth note: each bridge runtime-checks
`input.scheme !== this.scheme` and returns `{ok: false, reason:
'scheme_mismatch'}` rather than throwing. This catches the impossible
case where a refactor breaks the type-system narrowing — at runtime,
nothing 500s. (Same posture as T1.6's never-throw discipline.)

### D2 — Live-path quarantine: TWO layers + a script

Three independent enforcement mechanisms for FR-A4:

1. **Zod enum on `VerifyReq.scheme`** — `'siwe' | 'eip191'` only. The
   `dynamic_user_id` value is structurally inaccessible from the
   request body. Catches every HTTP request.

2. **Runtime `bridge.usableInLivePath` check** in the route handler
   BEFORE calling `bridge.verify()`. Returns 401
   `scheme_not_allowed_in_live_path` instantly. Catches future scheme
   additions that forget to re-audit Zod.

3. **Quarantine script** `scripts/check-dynamic-quarantine.sh` —
   greps live-path source for `import .* from "@dynamic-labs/..."`
   shapes (+ require/dynamic-import/re-export/package.json-dep). Catches
   the import surface itself.

The Zod layer is the operational firewall; the runtime check is the
contract anchor (it makes `usableInLivePath` a load-bearing flag, so
flipping it is a visible code-review event); the quarantine script is
the defense against the SDK landing for *any* reason (live or backfill).

The brief offered an ESLint rule as an alternative to the bash script.
Picked **bash** because:
- Zero new deps (ESLint isn't currently configured in this repo)
- Bash is grep + find — debuggable in one terminal command, no plugin authoring
- The check is wired to a `package.json` script (`bun run check:dynamic-quarantine`); operators get a one-word verification path

The script handles five import shapes:
- `import ... from "@dynamic-labs/..."` (named/default/namespace)
- `import "@dynamic-labs/..."` (side-effect)
- `import("@dynamic-labs/...")` (dynamic import)
- `require("@dynamic-labs/...")` (CJS interop)
- `export ... from "@dynamic-labs/..."` (re-export)

Plus package.json dependency-key detection (a violation even if not yet imported).

It does NOT trigger on prose mentions of `@dynamic-labs` in JSDoc or
comments — the regex anchors on `import`/`require`/`from` syntax, not
bare substring. The Dynamic bridge file itself contains the string
`@dynamic-labs` THREE times in its JSDoc explaining the discipline;
these are tested-negative in the synthetic-mirror suite (the
"prose-only" test plants the same shape and asserts exit 0).

### D3 — Audit-emit stays at the route layer

The bridges DO NOT emit audit events. The route handler's existing
`auth_signature_rejected` emit at `auth.ts:248-261` is preserved
verbatim; the bridge just returns `{ok: false, reason}` and the
route emits the audit row.

Why this matters: T1.6's audit chain has a strict shape (one event per
rejection, with the route holding the request context). If a bridge
emitted, we'd either double-emit (chain breaks) or have to plumb the
spine reference + request context through the bridge constructor.
Keeping emit at the route preserves the T1.6 contract verbatim.

The `VerifyRejectionReason` vocabulary in `credential-bridge.ts`
formalizes the shape the audit row will carry. T1.6's audit shape used
free-form reason strings from `verifySignature`; T1.7 extends that to
a typed union so future schemes can't introduce ad-hoc reason strings
without an explicit decision.

### D4 — Dynamic bridge: NO SDK, ever

The Dynamic bridge's input shape is `(dynamicUserId, walletAddress)`
— both are pre-extracted strings from a trusted backfill source row
(mibera-db's `dynamic_users` table per PRD §11 post-verify lock-ins).
The bridge does:

1. Shape-check the inputs (non-empty / hex format)
2. Lowercase the wallet
3. Return `{ok: true, walletAddress, linkedAccount: {provider:
   'dynamic_user_id', externalId: dynamicUserId}}`

That's it. No SDK call. No network I/O. No session validation.

This is correct because:
- The backfill source row IS the trust attestation. The T4.4 migration
  runbook authorizes the import; we're not re-validating per row.
- The live auth path uses SIWE/EIP-191 — Dynamic-session validation is
  by definition out of scope for identity-api.
- Adding the SDK would expand the supply-chain surface area (FR-A4
  was specifically about contracting it).

If you ever think you need the SDK in this file, the answer is "no" —
the bridge processes data, not sessions. Live-session validation was
the old (pre-T1.7) Dynamic role; T1.7 deletes that role.

### D5 — `linkedAccount` shape on Dynamic bridge success

`linkedAccount: { provider: 'dynamic_user_id', externalId: <input>
}` — the verbatim strings the T4.4 migration writes into
`linked_accounts`.

Decisions:
- `provider` is the literal string `'dynamic_user_id'` (matches PRD §4
  text). A unit test asserts the verbatim match because partitioning
  the `linked_accounts.provider` namespace with a typo would silently
  break resolve-by-account lookups.
- `externalId` is the input verbatim — no normalization, no
  case-folding. Dynamic IDs can be case-sensitive UUIDs; a round-trip
  test asserts mixed-case input survives unchanged.
- Wallet IS lowercased — matches spine storage normalization
  (`wallet_links.wallet_address` is stored lowercase everywhere).

The bridge does NOT compute `verified_at` — that's the engine's job
when the row is minted (the engine has a clock; the bridge doesn't).

### D6 — Bridges are singletons, not classes

`siweCredentialBridge`, `eip191CredentialBridge`, and
`dynamicCredentialBridge` are exported as `const`s, not constructor
classes. They're stateless — no config to inject, no per-instance
state to hold.

Pro: registry construction is trivial. The route's
`CREDENTIAL_BRIDGES` is a 3-line `Object.freeze` over the three
constants; nothing to `new`.

Future-proofing: if a bridge ever needs config (e.g., an ERC-1271
bridge needing an RPC URL), it can become a factory function returning
a `CredentialBridge` object — same interface, different construction
strategy. No interface churn required.

## Live-path enforcement at the route layer

The route's `usableInLivePath` check is at `src/api/routes/auth.ts:265-280`:

```typescript
// Step 3b: live-path quarantine (FR-A4 enforcement).
if (!bridge.usableInLivePath) {
  return unauthorized401(
    "scheme_not_allowed_in_live_path",
    "Credential scheme is not eligible for live-path verification",
  )
}
```

The Zod schema at `src/api/routes/auth.ts:160` is the first line of
defense:

```typescript
scheme: z.enum(["siwe", "eip191"]).default("siwe"),
```

`dynamic_user_id` is structurally inaccessible from the request body.
The runtime `usableInLivePath` check is unreachable via HTTP — but is
the structural contract anchor that proves the FR-A4 invariant: a
future PR adding `dynamic_user_id` to the Zod enum without auditing
its live-path eligibility would 401 instantly.

## Audit chain — unchanged from T1.6

T1.7 preserves T1.6's audit chain verbatim. The route handler still
emits `auth_signature_rejected` on a bridge rejection at the same
call site, with the same shape:

```typescript
await spine.writeAuditEvent({
  event_type: "auth_signature_rejected",
  user_id: null,
  actor: "self",
  payload: {
    reason: verifyRes.reason,        // ← from bridge VerifyRejectionReason
    wallet_address: wallet,
    scheme: body.scheme,
  },
})
```

The `reason` strings (`malformed_signature` / `signature_mismatch` /
`recover_error`) are width-subtype assignments from the SIWE/EIP-191
bridges' return shape — no information loss, same audit-row payload as
T1.6.

The T1.6 auth-flow test `wallet mismatch: challenge wallet A / verify
body wallet B → 401 wallet_mismatch + audit` passes verbatim after the
refactor, proving the audit shape is preserved.

## T4.4 integration sketch (the backfill bridge's consumer)

T4.4 (midi_profiles backfill migration) consumes the Dynamic bridge
directly — NOT via the route. Anticipated call shape:

```typescript
import { dynamicCredentialBridge } from '@freeside-auth/adapters'
import { resolveOrMintByWallet } from '@freeside-auth/engine'

// One-time migration walker
for await (const row of mibera_db_dynamic_users) {
  const result = await dynamicCredentialBridge.verify({
    scheme: 'dynamic_user_id',
    dynamicUserId: row.dynamic_user_id,
    walletAddress: row.wallet_address,
  })
  if (!result.ok) {
    // Skip malformed row; log to migration audit
    migrationLog.warn({ row_id: row.id, reason: result.reason })
    continue
  }

  await spine.withTransaction(async (tx) => {
    const { userId } = await resolveOrMintByWallet(tx, {
      walletAddress: result.walletAddress,
      actor: 'backfill-migration',
    })
    // Mint the linkedAccount row
    await tx.linkAccount({
      userId,
      provider: result.linkedAccount!.provider,    // 'dynamic_user_id'
      externalId: result.linkedAccount!.externalId, // verbatim ID
      actor: 'backfill-migration',
    })
  })
}
```

The bridge's `usableInLivePath: false` does NOT block this — that flag
gates the route handler, not direct callers. T4.4 calls the bridge
function directly; the live-path quarantine is irrelevant in that
context.

## Caveats / deferred

- **`verifySignature` is still exported from `@freeside-auth/adapters`** —
  the bridges are the canonical consumer now, but re-exporting the
  underlying primitive preserves T1.6 back-compat (e.g., tests that
  exercise the primitive directly). A future cleanup could mark the
  export `@deprecated` and route every caller through bridges.

- **`tsc --noEmit` is NOT in the `check` aggregate script** — vendored
  Hyper has 6 pre-existing baseline errors per T1.6 §K that block a
  clean tsc. Sprint-1.1 #5/#6 (the upstream Hyper patch) will let us
  re-add typecheck to the aggregate.

- **T4.4 (midi_profiles backfill migration) NOT in this task** — the
  bridge is built and tested as a unit; the migration consumer is a
  separate task. This task delivers the bridge in a state where T4.4
  can pick it up directly via the call shape sketched in §T4.4
  integration sketch.

- **ERC-1271 (contract wallets) still NOT supported.** Same posture as
  T1.6: SIWE/EIP-191 EOA-only. The bridge interface leaves room (a
  future `'erc1271'` scheme would just need its own bridge file +
  registry entry); no T1.7 plumbing required.

- **Bridges are STATELESS singletons.** Future bridges needing config
  (e.g., an RPC URL for ERC-1271) can switch to factory-functions
  returning a `CredentialBridge` object — same interface, different
  construction. No registry-shape churn.

- **The Dynamic bridge does NOT cross-check that the `dynamicUserId`
  is well-formed beyond shape constraints (non-empty, ≤256 chars).**
  This is intentional — Dynamic's ID format varies across product
  versions, and the backfill source attestation is the trust boundary.
  If a future migration needs stricter validation, it should be a
  caller responsibility (T4.4's migration runbook), not a bridge
  responsibility.

## §M · Cross-task contract for T4.4 / future bridges

T1.7 publishes the following surface for downstream tasks:

- **T4.4 (midi_profiles backfill migration)** consumes
  `dynamicCredentialBridge` from `@freeside-auth/adapters`. The
  migration walker calls `bridge.verify({scheme: 'dynamic_user_id',
  dynamicUserId, walletAddress})` per row; on `ok: true`, lands the
  `linkedAccount.provider/externalId` into `linked_accounts` via the
  engine's `linkAccount` (paired with `resolveOrMintByWallet` for the
  user/wallet bind in a `spine.withTransaction`). The bridge's
  `usableInLivePath: false` does NOT block direct callers — only the
  route's dispatch is quarantined.

- **Future credential schemes** (passkey / email / ERC-1271) follow
  the file-per-scheme pattern:
  1. Add `<scheme>` to `CredentialScheme` in `credential-bridge.ts`
  2. Extend `VerifyInput` union with the scheme-specific payload
     shape
  3. Author `credential-bridge-<scheme>.ts` implementing
     `CredentialBridge` (set `usableInLivePath` honestly!)
  4. Register in `CREDENTIAL_BRIDGES` at `src/api/routes/auth.ts`
  5. Update the Zod enum on `VerifyReq.scheme` if live-path
  6. The quarantine script auto-scans the new file (no script
     changes needed)
  7. Add the new bridge to `LIVE_PATH_ROOTS` only if the bridge file
     itself needs scanning (already covered for `packages/adapters/src/`)
