# T1.10 — `@freeside-auth/identity-client` SDK build notes

## Verdict

**Pattern B selected** (shared-schema imports, not codegen). Six commits land
the SDK + supporting refactor + tests. SDK is source-distributed per PRD §11
post-verify lock-ins — no npm publish, consumers vendor by source.

The load-bearing engineering target was **Sprint-1.1 follow-up #2** from the
T1.0 Hyper verdict: close the L6 path-typed-only gap by attaching per-route
input/output typing to the typed-client surface. This is now done — every
SDK method's request and response shape derives from the SAME Zod schema the
server validates against, via TypeScript inference. No codegen step.

## Layout decision: `packages/sdk/` vs `packages/identity-client/`

Chose `packages/sdk/` (short, conventional, mirrors the inventory-api building
which also has its SDK at `packages/sdk/`). Internal workspace name is
`@freeside-auth/identity-client`; external vendored alias is per-consumer
(typical: `@0xhoneyjar/identity` or `@/vendor/identity-client`). The README
documents the rename.

## Pattern A vs Pattern B — why B

| Aspect | (A) Codegen manifest | (B) Shared-schema imports |
|--------|----------------------|---------------------------|
| Build pipeline | new step: extract routes → emit `routes.ts` | none |
| Drift detection | regenerate + diff | inherent (single SoT) |
| Debug surface | step into generated code (often opaque) | step into the actual Zod schema |
| Bootstrap effort | ~1 sprint to write the codegen + tests | ~1 day to refactor + test |
| Power | richer (could emit OpenAPI-style helpers) | only what TypeScript can express |
| Vendoring friction | consumer needs the generated artifact | consumer copies plain `.ts` files |

**(B) wins for the consume model.** Vendored source distribution is the
canonical posture (PRD §11), and (B)'s output IS the canonical TypeScript
the consumer needs. (A)'s output is a separate codegen artifact that adds
friction to the vendoring step. The simpler design is the right one here;
(A) is the upgrade path if/when we need surfaces TypeScript can't express
(e.g., MCP tool manifests derived from the same routes).

## Engineering decisions

### 1. Where the shared schemas live

`packages/protocol/src/api/` — one file per route family
(`auth.ts`, `resolve.ts`, `profile.ts`, `link.ts`) + a barrel
`index.ts`. The protocol package is by design the home of sealed wire
shapes with zero runtime deps beyond Zod — exactly what the SDK needs to
vendor narrowly. Putting the schemas under `src/api/routes/*.ts` would
have forced the SDK to drag Hyper + spine + engine into the consumer's
tree.

The protocol package exports were extended:

- `@freeside-auth/protocol/api`         → barrel
- `@freeside-auth/protocol/api/auth`    → auth route schemas
- `@freeside-auth/protocol/api/resolve` → resolve + identity route schemas
- `@freeside-auth/protocol/api/profile` → profile + mibera stubs
- `@freeside-auth/protocol/api/link`    → link stub

### 2. `z.input` vs `z.output` split (load-bearing)

Initial cut used `z.infer` (= `z.output`) for caller-facing `*Req` types.
That made the SDK ergonomics hostile: `client.auth.challenge({
walletAddress: "0x…" })` failed to compile because `scheme` was marked
required (despite the schema's `.default("siwe")`).

Zod 4 separates these:

- `z.input<typeof Schema>`  → fields with `.default()` are OPTIONAL (the
  caller-facing shape)
- `z.output<typeof Schema>` → all defaults filled (the post-parse shape)
- `z.infer<typeof Schema>`  → alias for `z.output`

The fix:

```ts
// packages/protocol/src/api/auth.ts
export type ChallengeReq          = z.input<typeof ChallengeReqSchema>
export type ChallengeReqValidated = z.output<typeof ChallengeReqSchema>
```

- SDK uses `ChallengeReq` (input — caller-friendly).
- Server route handlers cast `c.body as ChallengeReqValidated` (output —
  defaults filled by Hyper's `.body()` validator).
- Both exported for advanced consumers.

This is the kind of subtle correctness issue (B) surfaces and (A) would
have hidden behind generated code.

### 3. Error class hierarchy

```
IdentityApiError                    (base; HTTP 4xx/5xx with envelope)
  ├── UnauthorizedError             (401)
  ├── ConflictError                 (409)
  ├── ValidationError               (400)
  ├── NotImplementedError           (501)
  └── (other 4xx/5xx fall through)

NetworkError                         (no response: DNS / socket / abort)
  └── extends IdentityApiError       (so catch-all works)
```

Each carries `status`, `code`, `message`, `requestId`, `envelope`,
`rawBody`. The `code` is the stable wire vocabulary
(`invalid_nonce`, `nonce_replayed`, `wallet_mismatch`) that consumers
switch on; `message` is human-readable and not load-bearing.

`NetworkError` extends `IdentityApiError` (status = 0 sentinel) so
consumers can write a single `if (e instanceof IdentityApiError)` catch
branch that covers every SDK failure. Specific narrowing
(`UnauthorizedError`, `ConflictError`) is layered on top.

The missing-JWT fast-fail in the transport throws `UnauthorizedError`
(not bare `IdentityApiError`) so the consumer's catch contract is
uniform: server-side 401 AND SDK-side fast-fail both surface as the same
class.

### 4. 404 → null convention for resolve/identity

`client.resolve.byWallet(addr)` returns `ResolveHitResp | null`, not
`Promise<ResolveHitResp>` that throws on 404. Same for `resolve.byAccount`,
`resolve.byNym`, `identity.get`.

Rationale: "user not found" is a routine answer for spine reads. Forcing
every caller to wrap in `try/catch` for the negative case would be
hostile. Other 4xx (400 malformed, 401 wrong auth) and 5xx propagate as
typed exceptions — those are NOT routine.

Implementation: a small `resolveOrNull()` helper that catches errors with
`status === 404` and converts to `null`. Uses duck-typing (`"status" in e`)
not class identity, so vendored copies with renamed exception classes
still benefit.

### 5. JWT injection

Three modes:

1. Static string: `createIdentityClient({ baseUrl, jwt: "abc.def.ghi" })`
2. Sync getter: `createIdentityClient({ baseUrl, jwt: () => store.token })`
3. Async getter: `createIdentityClient({ baseUrl, jwt: async () => await store.fetchToken() })`

Called on every authenticated request — the freshest token always wins.
This is the simplest pattern that supports rotating tokens without forcing
the consumer to recreate the client.

The `authHeader` option lets consumers override the bearer header (default
`"authorization"`); unusual but useful for reverse-proxied deployments
that strip the standard header.

### 6. Service-to-service token for `link.verifiedWallet`

A per-call opts arg (not client-construction) because:

- Most consumers DON'T call `link.verifiedWallet` (it's S2S-only — Sietch
  is the canonical caller).
- The S2S token has a different lifecycle from the end-user JWT.
- The header name is provisional (Sprint-1.x codifies). Per-call opts
  keep the surface flexible.

```ts
await client.link.verifiedWallet({...}, { serviceToken: "tok", serviceTokenHeader: "x-internal-key" })
```

Default header: `"x-service-token"`.

### 7. Stubs for unfinished endpoints

`profile.get` (T2.3), `mibera.dimensions` (T3.2), `link.verifiedWallet`
(T4.1) all have typed methods exposed today. They throw
`NotImplementedError(501)` at runtime. The typed signatures are correct;
consumers can write the call sites + catch-blocks now and the runtime
will deliver once the tasks land. No SDK call-site changes required when
the routes go live — only the response shapes tighten (loose `z.unknown()`
to specific shapes).

## Tests

### Trifecta

1. **`transport.test.ts`** (22 tests) — unit tests against a fetch shim.
   Covers URL construction, header composition, bearer JWT injection,
   2xx parsing, error class mapping, NetworkError path.

2. **`client.integration.test.ts`** (21 tests) — boots the real Hyper app
   on an ephemeral port + installs a mock spine. Drives every SDK method
   through actual fetch → route → handler → response. Asserts both
   runtime correctness AND TypeScript shape (via type-annotated
   assignments that wouldn't compile if the SDK return type were wrong).

3. **`types.type-test.ts`** (typechecked-only) — 11 `@ts-expect-error`
   directives proving the typed surface catches:
   - typo'd field names
   - wrong field types (number for string-typed field)
   - `dynamic_user_id` rejection in the live-path auth schemes
   - missing required fields
   - unknown extra fields
   - `ResolveHit | null` narrowing (forces explicit null check before
     accessing `.user_id`)
   - `IdentityResp` wallet/account field types
   - `linked_accounts[].provider` enum (no "twitter")
   - `link.verifiedWallet` `serviceToken` requirement

The `@ts-expect-error` directive only suppresses errors on the
immediately-following line. Multi-line object literals push the error to
the field line, leaving the directive "unused" — I had to collapse
several test cases onto one line so the directive pins to the correct
location.

## Cross-package zod version alignment

The protocol package previously declared `peerDependencies: zod ^3.23.0`,
which caused bun to install zod 3 into `packages/protocol/node_modules/`
while the root workspace ran zod 4. The type system mismatch broke
`z.infer<>` inference when importing protocol schemas from the root tree
(the imported schema was a zod-3 ZodObject, the root code expected zod-4
patterns).

Fixed by relaxing the peer dep to `zod ^3.23.0 || ^4.0.0`. Both protocol
AND sdk packages now share the root's single zod 4.4.3. External consumers
who pin zod 3 in their own tree still satisfy the peer range.

## Deferred / out of scope

- **Per-call JWT override** — current `jwt:` is client-scoped. Per-call
  override (for elevating to a service token mid-flow) would be a
  non-breaking SDK extension; not added until real demand surfaces.
- **JWKS-fetching helper** — consumers verifying tokens themselves need
  `/.well-known/jwks.json`; the SDK does not provide a fetcher today.
  Likely a separate vendored module (it would pull `jose` as a peer
  dep, which the current SDK doesn't).
- **Retry middleware** — the SDK is single-shot; retries are the
  consumer's responsibility. The transport is exported (`createTransport`)
  for advanced consumers who want to layer retry/backoff/OTel middleware
  around it.
- **Streaming** — no SDK method streams today. If a future endpoint does
  (e.g., a websocket-style notification feed), the SDK pattern extends
  cleanly: a new method on the client returning `AsyncIterable<T>` with
  the same Zod-derived typing.
- **OpenAPI client codegen** (Pattern A) — viable upgrade path if the
  number of routes grows beyond what (B) maintenance can handle, or if
  we need surfaces TypeScript can't express. Today: not needed.
- **ES256 swap implications** — the SDK doesn't validate JWTs itself; it
  just passes the bearer through to the server, which signs and verifies.
  The ES256 swap (Sprint-1.1 #3) is transparent to the SDK.
- **MCP tool surface alignment** — the SDK has no MCP awareness. MCP is
  a parallel consume organ (per PRD §11: "two-organ = source-distributed
  typed client + MCP discovery"). They share the same routes but emit
  through different stacks.
