## Discipline anchors (for next builders)

1. **NEVER edit `src/hyper/**` in-place.** It's vendored + SHA-pinned. Any
   patch goes through Sprint-1.1 follow-up #5 (vendored-patches log) so
   `bunx hyper update` knows what to re-apply.

2. **ALWAYS import `route` from `./auth`** (not from `@hyper/core`). The
   import-order discipline is what makes the L4 fix (installAuthMethod
   pre-installed) transitively guaranteed. Grep-detectable in review:
   `grep -r 'from "@hyper/core"' src/api/` should yield zero hits other
   than for response helpers / types.

3. **ALWAYS pair `session()` with `csrfGuard()`** via `applyWithSession()` —
   never call them separately. Verdict L5 lives or dies by this.

4. **NEVER omit `hostname:'0.0.0.0'` from `.listen()`.** Verdict L1 fix.
   Without it, Railway healthchecks fail and the deploy 503s. The bind is
   coded explicitly in `src/api/index.ts` line near the end.

5. **MCP exposure is per-route opt-in** via `meta.mcp.{title,description}`.
   Don't set those keys unless the route SHOULD be an agent-callable
   tool. /health is a counter-example (operations-internal, not a tool).

## Deviations from the verdict's auth.ts sketch

The verdict sketches `withSession()` returning a tuple meant to be
`.use(...withSession())`. **This doesn't work in Hyper 0.1.0** — Hyper's
`RouteBuilder.use(mw)` is unary (takes one Middleware, NOT a spread). I
preserved the tuple shape (`withSession(): [Middleware, Middleware]`) for
introspection / testing, and added a chain-helper `applyWithSession(builder)`
that calls `.use(sess).use(csrf)` on a builder. Usage:

```ts
// verdict sketch (doesn't compile):
const r = route.post("/v1/...").use(...withSession())  // ❌ "Expected 1 argument, got 2"

// T1.1 actual:
const r = applyWithSession(route.post("/v1/..."))      // ✓
  .body(Schema)
  .handle(...)
```

This is a presentational difference, not a semantic one — same
session + csrfGuard pairing rule, same L5 fix.
