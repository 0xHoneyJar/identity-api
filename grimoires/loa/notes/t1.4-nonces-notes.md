## §A · Layout choice: extend SpinePort vs new NoncePort

**Decision:** extend `SpinePort` (option a from the dispatcher's task).

**Why:**

1. **One port = one bounded context.** The `auth_nonces` table lives in the
   spine PG (same instance the rest of the spine writes to). A separate
   `NoncePort` would force a second adapter, a second singleton wiring, and
   a second audit-emit pairing — for a table that shares the same
   connection pool and audit chain as everything else.
2. **T1.5's `getSpine()` singleton already wires the connection.** T1.6's
   route handlers can call `getSpine().mintNonce(...)` exactly as they
   call `getSpine().resolveByWallet(...)` — no second module to import,
   no second test seam (`__setSpineForTest`) to install.
3. **The interface stayed small.** Two new methods, five new types. If
   nonce surface grows beyond ~5 methods (e.g., add `rotateScheme`, `dryRun`,
   per-tenant nonce sweep), extract `NoncePort` then — it's a mechanical
   split, the engine layer already hides the port from callers.

**Trade-off accepted:** `SpinePort` now has 11 methods (was 9). Past ~12-15
methods this would start to feel like an interface that wants splitting;
we'll revisit if cycle-2 adds more lifecycle methods (cleanup sweeps, nonce
rotation, etc.).

**Alternative refactor if needed later:** introducing `NoncePort` as a
proper subset doesn't require changes to existing callers — `SpinePort`
can `extends NoncePort` and the engine's type signatures remain
`SpinePort` even though they only call the nonce subset. Reversible.

## §B · DDL alignment finding (no 0003 needed)

T1.2's 0001 `auth_nonces` table is **bit-for-bit** what SDD §3.2 calls
for. Re-read both sources:

| SDD §3.2 spec | 0001 actual | Match? |
|---|---|---|
| `id UUID PRIMARY KEY DEFAULT gen_random_uuid()` | same | ✓ |
| `nonce TEXT NOT NULL UNIQUE` | same | ✓ |
| `wallet_address TEXT` (NULLable) | same | ✓ |
| `scheme TEXT NOT NULL CHECK (scheme IN ('siwe','eip191'))` | same | ✓ |
| `message TEXT NOT NULL` | same | ✓ |
| `expires_at TIMESTAMPTZ NOT NULL` | same | ✓ |
| `used_at TIMESTAMPTZ` | same | ✓ |
| `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()` | same | ✓ |
| `CREATE INDEX idx_auth_nonces_expires ON auth_nonces(expires_at) WHERE used_at IS NULL` | same | ✓ |

**Conclusion: no 0003 migration needed.** The shape is exactly the SDD
canonical re-expression of Sietch's `wallet_link_nonces` (SQLite → Postgres
translation per §3.2 grounding note).

The partial index `idx_auth_nonces_expires` is on `expires_at WHERE used_at
IS NULL` — this is what makes the FR-A1 expiry-sweep cheap once an
operator wires it (not part of T1.4). The atomic-consume path uses the
UNIQUE index on `nonce` for the `WHERE nonce = $1` lookup (B-tree, O(log
n), constant-time per-row at the storage layer).

## §C · Crypto discipline — nonce generation

Single source line, file `packages/adapters/src/postgres-spine-adapter.ts`
method `mintNonce`:

```typescript
const nonce = randomBytes(NONCE_BYTES).toString("base64url")
```

Where:
- `NONCE_BYTES = 32` (NFR-4 32-byte minimum; Sietch parity per
  `wallet-verification.ts` `NONCE_BYTES=32`).
- `randomBytes` is imported from `node:crypto` (Bun's node:crypto module
  is OpenSSL-backed; `randomBytes` is the CSPRNG primitive — NOT
  `Math.random`, NOT `Date.now`, NOT any pseudo-random).
- Encoding `base64url` yields 43 ASCII chars from 32 bytes with no
  padding (charset `A-Za-z0-9_-`). URL-safe, JSON-safe, cookie-safe.

**Constant-time comparison?** Considered and rejected as unnecessary. The
nonce lookup happens at the DB layer (`WHERE nonce = $1` against the
UNIQUE B-tree index) — that comparison is constant-time per-row at the
storage layer, and the response is the SAME (0 vs 1 rows) regardless of
whether the row exists, is used, expired, or scheme-mismatched. An
in-process `crypto.timingSafeEqual` would protect a different timing
surface that doesn't exist in this design.

The 4-class rejection discriminator (`unknown` / `used` / `expired` /
`scheme_mismatch`) is intentional: it gives the route layer the
information it needs to map to the SDD §5.6 error envelopes, AND it gives
an auditor counting rejection rates by class a useful signal (see §E
audit policy).

## §D · Atomic single-use — the load-bearing contract

FR-A1 promises: "single-use enforced by `used_at` (set on successful
verify)". Two concurrent verify calls for the same nonce MUST yield
exactly one success.

### The pattern

`PostgresSpineAdapter.consumeNonce` runs ONE statement:

```sql
UPDATE auth_nonces
   SET used_at = NOW()
 WHERE nonce = $1
   AND scheme = $2
   AND used_at IS NULL
   AND expires_at > NOW()
RETURNING message, wallet_address
```

This is RACE-SAFE because:

1. Postgres acquires a row-level lock for the duration of the UPDATE.
2. The first concurrent UPDATE that grabs the lock sets `used_at = NOW()`
   and returns 1 row.
3. The second UPDATE waits for the lock; when it acquires it, the WHERE
   clause `used_at IS NULL` now fails (because the first updater committed
   `used_at = NOW()` before releasing the row lock). The second UPDATE
   returns 0 rows.
4. The losing side then runs a SELECT to discriminate WHY (`used` /
   `expired` / `scheme_mismatch` / `unknown`).

### Why a TOCTOU read-then-write fails

The naive alternative:

```typescript
// HAZARDOUS — DO NOT USE
const row = await spine.findNonce(nonce);
if (!row || row.used_at || row.expires_at < now) return { ok: false, reason: ... };
await spine.markNonceUsed(nonce);
return { ok: true, message: row.message };
```

This has a TOCTOU race: two concurrent verify calls both pass the
`!row.used_at` check, both proceed to `markNonceUsed`, both UPDATE
succeed, both return `{ok: true}`. The attacker can replay one signature
to authenticate twice. This is exactly what FR-A1 forbids.

The UPDATE-RETURNING pattern collapses the read + write into one atomic
operation that the DB serializes via row-level locking — no application-
level coordination required.

### Test proof

Test files `packages/adapters/src/__tests__/postgres-spine-adapter-nonces.test.ts`:

| Test | What it proves |
|---|---|
| `consumeNonce concurrent race: 2 simultaneous calls → exactly ONE ok=true` | The basic race-safety proof: `Promise.all([consumeA, consumeB])` of the same nonce yields exactly 1 winner + 1 loser-classified-as-`used`. |
| `consumeNonce concurrent race at scale: 10 simultaneous calls → exactly 1 ok=true` | Regression-grade: 10-way race produces 1 winner + 9 losers all classified `used`. |

Both tests use `Bun.SQL`'s built-in connection pool — `Promise.all`
dispatches the queries onto distinct PG connections, which is the
prerequisite for the row-lock race to actually fire. Verified by running
the tests against postgres:18.1 — all 13 nonce-adapter tests pass
(§G run proof).

## §E · Audit policy (NFR-5)

Three event types emitted by the engine layer, all to `audit_events`:

| Event | When | Payload |
|---|---|---|
| `nonce_minted` | On successful `mintAuthNonce` | `{scheme, wallet_address, expires_at}` |
| `nonce_consumed` | On `consumeAuthNonce` ok=true | `{scheme, wallet_address}` |
| `nonce_rejected` | On `consumeAuthNonce` ok=false (each reason class) | `{scheme, reason}` |

**`user_id` is always `null` for all three** — the nonce flow is
pre-resolution by definition. The verify flow's user_id binding happens
DOWNSTREAM in T1.6's `resolveOrMintByWallet` step. The full audit chain
for /v1/auth/verify will be:

```
nonce_consumed (user_id=null)
  → user_minted   (user_id=NEW, if first-time wallet)
  → wallet_linked (user_id=NEW)
  → session_issued (user_id, T1.6 to define)
```

**Message text is NOT in any audit row.** Audit rows are dense and
queried by auditors; the full SIWE message is verbose, low signal, and
joinable later by `auth_nonces.nonce` if a specific challenge needs
forensic recovery. The scheme + wallet + expires_at carry the value.

**Rejection `reason` is in payload** so an auditor can `GROUP BY
payload->>'reason'` to count rates per class:
- spike in `unknown` → guessing attack / nonce table tampering
- spike in `used` → replay attempt detection
- spike in `expired` → client clock drift OR honest-user-took-too-long
- spike in `scheme_mismatch` → client confusion / scheme-migration regression

**Audit row count from end-to-end demo (§G):** 10 rows total for 4 mints
+ 2 ok-consumes + 4 reject-consumes. Wire-side confirmed.

## §F · T1.6 integration contract

The endpoint shapes T1.6 will build, derived from SDD §5.2:

### POST /v1/auth/challenge (FR-A1)

```typescript
import { mintAuthNonce } from "@freeside-auth/engine"
import { getSpine } from "../api/spine"

handler: async (req) => {
  const { walletAddress, scheme } = req.body  // { walletAddress?: "0x..", scheme: "siwe"|"eip191" }
  const expiresAt = new Date(Date.now() + 300_000)  // 5 min ahead (echoed into SIWE message)
  const message = scheme === "siwe"
    ? buildSiweMessage(walletAddress, /* placeholder until known */, domain, expiresAt)
    : buildEip191Message(walletAddress)
  // NB the SIWE builder needs a nonce IN the message — call mintAuthNonce
  // FIRST, then build the message with the returned nonce, then... actually
  // the cleanest path is: mint with a *placeholder* message, build the real
  // message using the returned nonce, then UPDATE the row's message column.
  // OR: refactor mintAuthNonce to accept a message-builder closure that
  // takes the random nonce and returns the message. T1.6 picks one.
  const { nonce, expires_at } = await mintAuthNonce(getSpine(), {
    scheme,
    message,
    walletAddress: walletAddress ?? null,
    actor: "self",
  })
  return { nonce, message, expiresAt: expires_at }
}
```

**Note for T1.6:** the SIWE flow embeds the nonce IN the message, which
creates a chicken-and-egg with `mintAuthNonce` (which generates the
nonce). Two viable patterns:

- **Two-write pattern:** mint with a placeholder message, build real
  message with the returned nonce, `UPDATE auth_nonces SET message=...
  WHERE nonce=...`. Costs one extra write but keeps `mintAuthNonce` simple.
- **Builder closure pattern:** refactor `mintAuthNonce` to accept
  `(nonce: string) => string` instead of `message: string`. The adapter
  generates the nonce, calls the closure to build the message, then
  INSERTs both. One write. T1.4 left the simpler `message: string`
  shape because it serves the EIP-191 path verbatim; T1.6 picks the
  refactor if SIWE wants it.

### POST /v1/auth/verify (FR-A2)

```typescript
import { consumeAuthNonce, resolveOrMintByWallet } from "@freeside-auth/engine"
import { getSpine } from "../api/spine"
import { SignatureVerifier } from "../auth/verifier"  // T1.6 vendor

handler: async (req) => {
  const { nonce, signature, walletAddress, scheme } = req.body
  const spine = getSpine()
  const consumed = await consumeAuthNonce(spine, {
    nonce,
    expectedScheme: scheme,
    actor: "self",
  })
  if (!consumed.ok) {
    // map reason → SDD §5.6 envelope
    const code = consumed.reason === "used" ? "CHALLENGE_USED" : "CHALLENGE_EXPIRED"
    return res.status(401).json({ error: { code, message: "...", requestId } })
  }
  const valid = await SignatureVerifier.verifyAddress(
    consumed.message, signature, walletAddress
  )
  if (!valid) {
    return res.status(401).json({ error: { code: "INVALID_SIGNATURE", ... } })
  }
  const { userId } = await resolveOrMintByWallet(spine, {
    walletAddress, actor: "self"
  })
  const session = await mintJwt({ sub: userId, wallet: walletAddress })
  res.setCookie("session", encryptSession(session), { httpOnly: true, sameSite: "Lax", secure: true })
  return { userId, session }
}
```

T1.6 reads `consumed.message` (the verbatim string signed by the wallet)
and `consumed.wallet_address` (the hint stored at mint time — may be
null in pre-bind flows).

## §H · Open questions / future work

- **SIWE message-building chicken-and-egg** (mentioned in §F). T1.6 may
  want a `buildMessage: (nonce) => string` closure shape on `mintAuthNonce`.
  Reversible refactor; doesn't affect T1.4's adapter or audit policy.
- **Expiry sweep cron.** The `idx_auth_nonces_expires` partial index is
  there to make `DELETE FROM auth_nonces WHERE expires_at < NOW() - INTERVAL
  '1 day'` cheap; no sweep is wired in T1.4. Operator decision when/whether
  to add (the table grows ~slowly given default TTL 300s + most nonces are
  consumed).
- **Per-wallet rate limit on /challenge.** Not part of T1.4 (route layer).
  When T1.6 wires the endpoint, an `INSERT` rate limit per `wallet_address`
  would defend against a churn attack that fills up the nonce table.
  Currently unbounded (within the size of `users` * `RPS`).

## §I · Deviations / caveats

- **Tests skip without `TEST_DATABASE_URL`.** Same pattern as
  `postgres-spine-adapter.test.ts` and `migrate.test.ts`. The CI
  acceptance is that the suite passes either way; the DB-backed proofs
  run when an operator/CI provides a scratch URL.
- **No 0003 migration was needed** — T1.2's 0001 `auth_nonces` table
  matches SDD §3.2 exactly. Branch commits reflect that (no `chore(migrate)`
  commit landed).
- **MockSpine seams in `routes.test.ts` + `resolve-spine.test.ts` got
  two trivial new stub methods** (`mintNonce` / `consumeNonce`) so the
  test files still typecheck against the extended `SpinePort`. Zero
  behavior change for tests that don't exercise nonces.
- **Bun.SQL date column type quirk:** when reading `TIMESTAMPTZ` columns,
  Bun.SQL returns either `Date` (newer drivers) or `string` (older) — the
  adapter normalizes via `expiresAt instanceof Date ? .toISOString() :
  new Date(value).toISOString()` for a stable ISO 8601 contract.
