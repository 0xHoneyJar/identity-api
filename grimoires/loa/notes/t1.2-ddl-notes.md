# T1.2 — author the spine SoR migration

T1.2 lands the central resolution spine — the DDL that makes identity-api a
*writer* (D1 / Q1 ratified writer-graduation), not a read-only adapter
over `midi_profiles`. Everything in this migration realizes PRD v3.0 §4.2
+ SDD §3.2 verbatim, with intentional, documented deviations called out
below.

## Scope (what landed)

- `packages/adapters/src/migrations/0001_init_spine.up.sql` — **7 tables** + all
  indexes, FK-safe ordered:
  - `users` (UUID PK, `primary_wallet` text NULL, timestamps)
  - `wallet_links` (multi-chain `chain_ids text[]`, soft-unlink via
    `unlinked_at`, **2 partial-unique indexes** — active-address uniqueness
    + one-primary-per-user enforcement)
  - `linked_accounts` (provider CHECK in `('discord','telegram','dynamic_user_id')`,
    composite PK `(provider, external_id)`)
  - `worlds` (PK `world_slug`, seed deferred to deploy per OQ-5)
  - `world_identity` (composite PK `(user_id, world_slug)`, UNIQUE `(world_slug, nym)`)
  - `audit_events` (NULLABLE `user_id`, soft-pointer — no FK — survives user
    deletion per NFR-5)
  - `auth_nonces` (FR-A1 challenge/response lifecycle, single-use via
    `used_at`, partial idx on `expires_at WHERE used_at IS NULL`)
- `packages/adapters/src/migrations/0001_init_spine.down.sql` — full reversal
  in reverse FK order; leaves DB clean (only `schema_migrations` + pgcrypto
  remain — both shared infrastructure, intentionally preserved)
- `packages/adapters/src/migrate.ts` — minimal Postgres migration runner
  (Bun built-ins only — `Bun.SQL` + `node:fs` + `node:path`). Verbs `up`,
  `down`, `status`. Tracks state in `schema_migrations(version, applied_at)`.
- `scripts/migrate.ts` — thin CLI shim that imports and delegates to the
  runner. Both paths work; `scripts/migrate.ts` is the user-facing entry
  per the T1.2 prompt.
- `package.json` — added `migrate` / `migrate:down` / `migrate:status`
  scripts.
- `packages/adapters/src/__tests__/migrate.test.ts` — round-trip integration
  test (gated on `TEST_DATABASE_URL`; skips gracefully when unset). 6
  tests, 53 assertions.

## Placement decision — why `packages/adapters/src/migrations/`

The SDD (§2.2) lists `drizzle-kit` as a "pin at Phase 1" option for the
migration tool. Whichever tool wins, the **SQL files** themselves stay
useful (drizzle-kit consumes raw SQL too). Three reasons to colocate them
under `packages/adapters/src/migrations/`:

1. The existing PG seam already lives here (`postgres-split-adapter.ts`).
   Migrations and adapter code share a bounded context — both are I/O at
   the L4 boundary (P3 Execution plane per ADR-008). Pure logic stays in
   `engine`/`protocol`.
2. The runner can resolve its migrations dir relative to its own source
   file (`defaultMigrationsDir()`), so invocations from repo root **or**
   from inside the package both work without env vars.
3. If/when drizzle-kit lands, the swap is local to one workspace —
   `packages/adapters/drizzle.config.ts` + a meta directory; no other
   package needs to know.

The alternative — `migrations/` at the building root — would have been
fine too, but would couple to the runner's process.cwd() and bleed into
the building-root surface that today is reserved for `src/`,
`packages/`, `scripts/`, `docs/`, etc.

## Type choice — UUID, not text-ULID

The T1.2 prompt left this open ("`text` ULID or `uuid` if the SDD
specifies — check"). The canonical SDD §3.2 is unambiguous: `UUID PRIMARY
KEY DEFAULT gen_random_uuid()` + `CREATE EXTENSION pgcrypto`. The PRD's
§10 reconciliation block mentions "ULID `user_id`" as the building's old
2026-04-30 plan; the operator-ratified v3.0 PRD §4.2 sketch + the SDD
§3.2 realization both use UUID. **UUID it is.** Worth noting that the
protocol package's `UserSchema.sub` is `z.string()` (not strictly UUID),
which renders fine over a UUID column — Postgres serializes UUID as text
to JSON on the wire.

## Provider CHECK enum — 3, not 5

T1.2 prompt mentioned `['discord','telegram','dynamic_user_id','email','twitter']`.
SDD §3.2 ships only `('discord','telegram','dynamic_user_id')` in v1.
PRD §2.2 explicitly defers passkey / email / social-OAuth (`additive
bridges later (the credential-passkey/-seedvault stubs already reserve
this)`); email and twitter handles are out-of-scope for v1.

**Followed SDD.** When email/twitter land later, the CHECK constraint is
a one-line `ALTER TABLE linked_accounts DROP CONSTRAINT … ADD
CONSTRAINT … CHECK (...)` in a future migration. Doing it now would
silently broaden the v1 contract beyond what the resolution spine is
designed to handle.

## What I did NOT add (per the T1.2 hard constraints)

- **No primary-wallet integrity trigger** — that's T1.3 (migration 0002,
  bead arrakis-ca51). The partial-unique index
  `uq_wallet_links_one_primary_per_user` IS in 0001 because the SDD §3.2
  comment makes it explicit ("The partial index is the hard guarantee
  (FR-R5 'exactly one primary'); the trigger is the convenience"). The
  hard guarantee ships with the spine; the convenience ships with 0002.
- **No spine write logic** — that's T1.5 resolve (bead arrakis-232n).
- **No worlds seed** — SDD §13 OQ-5 default is "seed at deploy from
  freeside-worlds registry"; migration 0001 is structural only.
- **No `NULLS NOT DISTINCT`** — the T1.2 prompt mentioned it for natural-
  key UNIQUEs "where the SDD specifies it." The SDD §3.2 does NOT
  specify it for any of the indexes. All natural-key UNIQUE indexes
  here either have NOT NULL components (`linked_accounts` composite PK,
  `world_identity.UNIQUE(world_slug, nym)`), or are partial indexes whose
  WHERE clause already filters NULLs (`uq_wallet_links_active_address`,
  `uq_wallet_links_one_primary_per_user`). Adding `NULLS NOT DISTINCT`
  would be no-op (no nullable columns participate) or contradict the
  partial-index semantic. **Not added.**

## Open questions for review

1. **Provider enum bounded growth.** Email + Twitter are deferred per PRD
   §2.2 / SDD §3.2. The CHECK constraint here is restrictive. If you'd
   prefer to ship a more liberal CHECK now to avoid the ALTER later,
   say so — but the SDD is explicit on the v1 list, so I followed it.
2. **`audit_events.user_id` as soft pointer (no FK).** SDD §3.2 has the
   column as `UUID` without REFERENCES; I followed that and documented
   the rationale (pre-resolution conflicts have no user; audit must
   survive user deletion). Confirm this matches your intent — the
   alternative is `REFERENCES users ON DELETE SET NULL`, which would
   preserve audit on delete but require the user to exist at write time.
3. **`scripts/migrate.ts` as the user-facing CLI vs `packages/adapters/src/migrate.ts`
   directly.** Both work today. Did you want only one entry point? I
   kept both because the runner's `main()` is `import.meta.main`-gated
   (no double-side-effect when imported by tests/script).
4. **State table strategy.** `schema_migrations` is intentionally NOT
   dropped on `down` — it's the runner's own ledger, not part of the
   spine. If you want full erasure on `down 0001`, that would require
   a separate `migrate reset` verb (deliberately out of scope here).
5. **Drizzle-kit pin.** SDD §2.2 flags it as a "pin at Phase 1" option.
   This runner defers that pin (zero-dep, source-distribution discipline
   per T1.0/T1.1 vendor posture). If you want to commit to drizzle now,
   the SQL files port directly and `scripts/migrate.ts` is a 30-line
   shim around `drizzle-kit migrate`.
