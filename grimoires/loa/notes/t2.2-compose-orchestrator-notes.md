## 2. Per-source timeouts (confirmed)

Per SDD ¬ß6.2 + the runbook spec:

| Source | Default | Constant |
|--------|---------|----------|
| inventory-api | **500ms** | `DEFAULT_INVENTORY_TIMEOUT_MS` |
| score-api | **300ms** | `DEFAULT_SCORE_TIMEOUT_MS` |
| codex (Mibera batch) | **400ms** | `DEFAULT_CODEX_TIMEOUT_MS` |

Worst-case sequential time ‚âà `max(T_inv, T_score) + T_codex = 500 + 400 = 900ms`
ceiling; the p95 target of 800ms (NFR-1) assumes typical sub-timeout
latency. All three are configurable via `ComposeProfileOpts` for tests + a
future operator-tuned production override.

The orchestrator OWNS the timeouts (per runbook). Adapters receive an
`AbortSignal` opaquely and forward it to `fetch`; the shared
`federationHttpCall` helper classifies abort vs network_error via the
`signal.aborted` check (see T2.1 build notes ¬ß4).

## 3. Circuit-breaker shape (per SDD ¬ß6.4 ‚Äî "lightweight v1")

SDD ¬ß6.4 doesn't pin exact numbers; the orchestrator picks:

| Parameter | Default | Why |
|-----------|---------|-----|
| `failureThreshold` | **5** | Conservative; tolerates noise but trips on sustained outage. |
| `rollingWindowMs` | **60_000 (1 min)** | Failures older than this drop off; prevents a slow trickle from tripping. |
| `cooldownMs` | **30_000 (30s)** | Half the rolling window ‚Äî short enough to recover quickly when upstream comes back, long enough to avoid hammering during sustained outage. |

State machine:

```
  ‚îå‚îÄ‚îÄ‚îÄ recordSuccess (resets counter)
  ‚ñº
‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê  N failures within rolling window
‚îÇ closed ‚îÇ ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚ñ∫ ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê
‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò                                   ‚îÇ open ‚îÇ
   ‚ñ≤                                         ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
   ‚îÇ recordSuccess                              ‚îÇ
   ‚îÇ                                            ‚îÇ cooldownMs elapsed
   ‚îÇ                                            ‚ñº
‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê    recordFailure   ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê

```

Per-source independence: one breaker per source (not one shared) so one
source's outage doesn't open the breaker for the other two.

**In-memory state**: per NFR-3 the single Railway instance makes shared
state trivial ‚Äî no Redis. Breakers are module-level instances in T2.3's
route-handler bootstrap (the orchestrator takes them via `deps.breakers`).

**Score 404 is healthy**: `not_found` reasons (typical for score-api on
unindexed wallets per ScorePort docstring) call `recordSuccess()` ‚Äî a 404
means the upstream is healthy enough to respond with a typed error. Only
`timeout` / `upstream_5xx` / `parse_error` / `network_error` / `unauthorized`
tick the failure counter.

**`circuit_open` results NOT recorded back**: they're synthesized internal
short-circuits, not external health data. Recording them would loop the
breaker open forever once it tripped.

## 4. Codex two-phase confirmed (pattern B)

Per SDD ¬ß6.1 topology: `INV ‚Üí CX` belt ‚Äî codex needs the wallet's Mibera
tokenIds, which come from inventory's holdings. Pattern A (single
allSettled fan-out treating codex as parallel) would require codex to
accept a wallet address + do its own filtering, which it doesn't. Pattern
B (sequential) is the only correct shape:

```
Phase 1 (parallel via Promise.all):
  inventory.getHoldings(walletAddress) ‚îÄ‚îÄ‚îÄ‚îê
                                          ‚îú‚îÄ‚îÄ await both
  score.getScore(walletAddress) ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò

Phase 2 (conditional, runs IFF Phase 1 inventory succeeded
                                AND walletAddress has Mibera tokens):
  tokens = extractMiberaTokens(holdings.holdings)
  if tokens.length > 0:
    codex.getMiberaTraits({tokenIds: tokens})
```

Skip semantics:
- **Holdings degraded ‚Üí codex SKIPPED, no extra degraded[] entry.** The
  inventory degraded entry already explains why codex didn't run. Adding
  `"codex:no_input"` would double-count the same root cause.
- **Holdings ok but no Mibera tokens ‚Üí codex SKIPPED, no degraded entry.**
  The wallet simply holds no Mibera; that's a normal-shape response,
  not an error.
- **Holdings ok + Mibera tokens + codex fails ‚Üí codex degraded entry**
  (`"codex:timeout"`, `"codex:upstream_5xx"`, etc.).

## 5. Promise.all (NOT Promise.allSettled) ‚Äî discrepancy with bead title

The bead title says "Compose fan-out orchestrator ‚Äî per-source timeouts +
lightweight in-memory circuit-breaker + `degraded[]` invariant" and PRD ¬ß4.5
FR-P2 text says "Fan-out with per-source timeouts + `Promise.allSettled`".
SDD ¬ß6.3 pseudocode also shows `Promise.allSettled`.

**Decision: `Promise.all` is correct per T2.1's design.** Reason:

T2.1 chose the discriminated-union `FederationResult<T> = {ok:true, data} |
{ok:false, reason}` contract on every federation port specifically so the
T2.2 orchestrator could use `Promise.all` cleanly. Every adapter resolves
its promise no matter what ‚Äî failures surface as `ok: false`, not
rejections. With this contract:

```ts
const [holdings, score] = await Promise.all([
  inventory.getHoldings(...),
  score.getScore(...),
])
// holdings + score are both FederationResult<...>; no try/catch needed.
```

vs the `allSettled` shape, which would wrap each into
`{status: 'fulfilled'|'rejected', value|reason}` requiring an unwrap:

```ts
const [holdingsRaw, scoreRaw] = await Promise.allSettled([...])
if (holdingsRaw.status === 'fulfilled') { ... } else { /* unknown */ }
```

`allSettled` would be necessary if adapters could throw ‚Äî but they don't.
The PRD/SDD `allSettled` language is a vestige of an early design before
T2.1 sealed the result-type contract; the union design supersedes.

T2.1's build notes ¬ß2 documents this explicitly:

> "Discriminated union is more T2.2-friendly (no try/catch in the
> fan-out)." Confirmed during the build: T2.2's `Promise.allSettled`
> always wraps an inner throw into `{ status: 'rejected', reason: unknown }`
> ‚Äî losing typed-error knowledge.

This note is the artifact recording the design supersession; the PRD/SDD
text can be updated in a follow-up doc pass (no functional impact).

## 6. T2.3 integration sketch (`/v1/profile` route handler)

T2.3 wires `composeProfile` into the 501-stub `getProfile` route handler
at `src/api/routes/profile.ts`. Sketch:

```ts
// src/api/routes/profile.ts (T2.3 wire-up)
import { getSpine } from "../spine"
import { getInventory } from "../inventory"   // T2.1 singleton
import { getScore } from "../score"           // T2.1 singleton
import { getCodex } from "../codex"           // T2.1 singleton
import { composeProfile, CircuitBreaker } from "@freeside-auth/engine"

// Module-level breakers (in-memory state, single-Railway-instance).
// Created once per process; survive across requests until restart.
const breakers = {
  inventory: new CircuitBreaker(),
  score: new CircuitBreaker(),
  codex: new CircuitBreaker(),
}

export const getProfile = route
  .get("/v1/profile")
  .body(ProfileQuerySchema)
  .handle(async (req) => {
    const q = req.body  // {world, userId?, wallet?}
    const input = q.userId
      ? { userId: q.userId }
      : { walletAddress: q.wallet! }
    try {
      const profile = await composeProfile(
        {
          spine: getSpine(),
          inventory: getInventory(),
          score: getScore(),
          codex: getCodex(),
          breakers,
        },
        input,
        { actor: req.session?.userId ?? "system" },
      )
      return jsonResponse(200, profile)
    } catch (err) {
      const msg = (err as Error).message
      if (msg === "user_not_found" || msg === "wallet_not_resolved" || msg === "primary_wallet_missing") {
        return jsonResponse(404, { error: "not_found", reason: msg })
      }
      throw err  // real spine I/O ‚Üí 5xx via global error handler
    }
  })
```

T2.3's scope:
- Add the route-level breaker singletons (module-level).
- Map orchestrator throws to 404 envelopes (3 known throw kinds).
- Validate `q.world` against the worlds registry (per FR-P1 the
  `world_slug` is a query param; T2.2 ignores it because all 3 federation
  sources are world-agnostic in v1 ‚Äî Mibera-specific world filtering is a
  T3+ surface).
- Move `ProfileQuerySchema` from `.body()` to `.query()` if Hyper has
  shipped the builder by then; otherwise keep `.body()` (per existing
  comment at `src/api/routes/profile.ts:32`).

## 7. Caveats + deferrals

### `extractMiberaTokens` Mibera collection-detection

Resolved: matched against the lowercased Mibera contract address
`0x6666397dfe9a8c469bf65dc744cb1c733416c420` (Berachain chainId 80094).
Source: `~/Documents/GitHub/inventory-api/src/inventory.ts:18`
(`MIBERA_CONTRACT`). Case-insensitive comparison so checksum or lowercase
forms from upstream both match.

Edge case: if upstream-inventory ships a multi-chain Mibera (e.g., a
bridged copy on another chain), the chainId predicate would need to
loosen. Not in v1 scope; codex's wire schema is currently
Berachain-Mibera-only.

### inventory-api known sonar-ownership gap

Per `~/Documents/GitHub/inventory-api/docs/sonar-ownership-gap.md`, live
mode currently returns real `tokenCount` with `tokenIds: []` (the per-token
ownership index isn't published yet). `extractMiberaTokens` returns `[]`
for those, codex is skipped, no degraded entry is emitted. When upstream
closes the gap, the codex compose lights up automatically.

### Cap at 100 tokenIds per request

Codex's `POST /v1/mibera/batch` wire schema caps at 100 tokenIds
(CodexGetMiberaBatchReqSchema). `extractMiberaTokens` truncates to the
first 100. Wallets holding >100 Mibera get a partial codex result; UX
layer can render a "more available" indicator if it cares. Paginated
batch is a T3+ refinement.

### Breaker shared-state across processes

Per NFR-3 single-Railway-instance. If we ever scale to multi-instance,
the in-memory breaker state would be per-process ‚Äî not problematic in
itself (each process protects itself), but the cluster-wide health
signal would be N times noisier. Either accept that, or move to Redis
breaker (out of v1 scope).

### Mock-port import path

The compose-profile.test.ts imports MockInventoryPort/MockScorePort/
MockCodexPort via relative path (`../../../adapters/src/__tests__/*`)
instead of via the `@freeside-auth/adapters/__tests__/*` subpath. Reason:

1. Adding the subpath to `packages/adapters/package.json::exports` would
   expose the mock files as a runtime entry-point (they're test-only).
2. Adding `@freeside-auth/adapters` as an engine devDependency would
   create a circular dep (adapters ‚Üí engine already exists).

The relative path is the cleanest of the bad options. If we ever extract
the mock ports into a separate `@freeside-auth/test-fixtures` workspace
package, the imports update; the tests don't change.
