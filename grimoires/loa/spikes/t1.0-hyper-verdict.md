# T1.0 — Hyper runtime de-risk verdict

> **VERDICT: GO-WITH-CAVEATS**
>
> Hyper does deliver one-definition → {runtime + OpenAPI 3.1 + typed client + MCP} from the same source, and the JWT / encrypted-cookie / CSRF plugins are all real, work end-to-end, and follow secure-by-default patterns (32-byte secret minimum, HttpOnly + SameSite=Lax + Secure cookies, double-submit CSRF). The "distributed as source" claim is also real and tooled (SHA-pinned via `hyper.lock.json`, drift via `hyper diff`, registry-backed updates).
>
> The caveats are real but bounded: (a) the "typed" client is path-typed only — request bodies and responses are `unknown`; (b) the CLI `hyper openapi` emit doesn't pick up Zod schemas (runtime `/openapi.json` does); (c) the vendored source has ~10 type errors against current `@types/bun` / strict TypeScript that don't affect runtime; (d) the codegen has UX bugs (output `<path>` becomes a directory, the `hyper openapi <path>` positional is destructive as the operator warned). None block the SDD-described architecture, but each adds 0.5–1 sprint of papercut friction. **Build identity-api on Hyper. Budget Sprint 1.1 to land a thin in-house `withSession()` / `withAuth()` wrapper that restores body+response typing, and capture a vendored-source patch list.**

## VERDICT: GO-WITH-CAVEATS

**Two strongest pieces of evidence:**

1. **The one-def claim holds for the four artifacts.** A single `route.post("/v1/auth/challenge").body(ChallengeReq).meta({mcp:{...}}).handle(...)` definition produces all four surfaces on this machine, today, against the pinned commit:
   - **Runtime:** `curl POST /v1/auth/challenge` → 200 with structured body; `curl POST {bad-wallet}` → 400 with structured `validation_failed` and per-field issue paths.
   - **OpenAPI 3.1 (runtime):** `/openapi.json` returns `{openapi:"3.1.0", paths:{"/v1/auth/challenge":{post:{requestBody:{content:{"application/json":{schema:{type:"object", properties:{walletAddress:{type:"string"}, scheme:{enum:["siwe","eip191"], default:"siwe"}}, required:["walletAddress","scheme"]}}}}}}}}` — Zod schema → JSON Schema lossless.
   - **Typed client (CLI):** `bunx hyper client gen-client.ts` emits `api.post.v1.auth.challenge` leaf with path/method typed.
   - **MCP (CLI audit):** `bunx hyper mcp --audit` lists `POST /v1/auth/challenge` as an exposed tool with its `meta.mcp.description` verbatim. Per-route opt-in (5/7 non-MCP routes correctly hidden).

2. **The auth plugin trio works end-to-end with the secure-by-default ergonomics the SDD assumes.** JWT (HS256 verified, `sub` extracted to `ctx.user`, `.auth()` sugar gates the route, malformed Authorization → 401 with `WWW-Authenticate: Bearer realm="hyper"`). Encrypted-cookie session (`idapi_sess=<id>.<HMAC-sig>; HttpOnly; SameSite=Lax; Secure; Max-Age=604800`). CSRF double-submit (mutating POST without `X-CSRF-Token` → 403 `csrf_token_mismatch` with a `why`/`fix` payload; mutating POST with the token → 200 + session payload preserved). All three composed in 3 `.use()` lines per route.

## What was rough (gotchas hit — operator's landmines confirmed + new ones surfaced)

### Operator-flagged landmines — all three CONFIRMED

| # | Operator's warning | Confirmation in this spike |
|---|--------------------|----------------------------|
| **L1** | `.listen()` binds `localhost` off-prod → healthcheck failure | **Verified.** First run with `.listen(Number(process.env.PORT))` logged `listening on http://localhost:<port>`. Replacing with `.listen({port, hostname:"0.0.0.0"})` logs `listening on http://0.0.0.0:<port>` ✓. The exact code line that defuses it is `spike/src/app.ts:163-166`. |
| **L2** | `hyper openapi <path>` is DESTRUCTIVE — treats positional as OUTPUT not input | **Verified + WORSE than stated.** `bunx hyper openapi` (no arg) prints to stdout. `bunx hyper openapi spike-openapi.json` writes the file (overwriting if present). I did not test pointing it at a source file, but the CLI usage `openapi [out]` and the help text confirm operator's framing. **Extension:** `bunx hyper client <out>` creates `<out>/` as a **DIRECTORY** containing `client.ts` + `client.d.ts` — so `bunx hyper client client.gen.ts` produces `client.gen.ts/client.ts` and `client.gen.ts/client.d.ts`. Easy to miss; `gitignore` should target `<out>/` not `<out>` to catch this. |
| **L3** | Hyper 0.1.x is bleeding-edge — may need in-place patches to vendored `src/hyper/*` + `DOM, DOM.Iterable` in tsconfig `lib` for `@types/bun` skew | **Verified.** Added `DOM, DOM.Iterable` to tsconfig `lib` (fixed ~3 errors in `client/transport.ts` re `BodyInit`). Remaining typecheck errors in vendored Hyper source after that fix: ~10, spread across `auth-jwt/jwt.ts` (`crypto.subtle.importKey` overload pickiness re `Uint8Array<ArrayBufferLike>` vs `BufferSource`), `openapi-zod/index.ts:65` (ZodDef cast mismatch), `openapi/generate.ts:119` (spread types), `session/index.ts:117` (implicit-any in callbacks), `core/app.ts:187` (URL not assignable to fetch RequestInfo). None affect runtime behavior — Bun runs all of this happily — but a `tsc --noEmit` will be noisy and CI gating on typecheck will need a `tsc --skipLibCheck` plus possibly excluding `src/hyper/**` from strict checking until upstream fixes. |

### New landmines discovered in this spike

| # | Issue | Workaround used in spike | Impact on identity-api |
|---|-------|--------------------------|------------------------|
| **L4 (new)** | `.auth()` is added to `RouteBuilder.prototype` at plugin-construction time by `authJwtPlugin`. Module-eval `route.get(...).auth()` definitions race the plugin boot and 500 with `route.get("...").meta({...}).auth is not a function` | Eagerly install at module top via `installAuthMethod(authJwt(config))` BEFORE any `.auth()` is used in a route def (`spike/src/app.ts:22-25`). Production: codify in a single `src/auth.ts` that exports a pre-installed `route` re-export | Real impact: every file that defines an `.auth()` route must import in dependency order. A `src/auth.ts` re-export pattern + a small unit test that ensures `RouteBuilder.prototype.auth` is set before route registration would catch this. |
| **L5 (new)** | `csrfGuard()` only issues the csrf cookie on RESPONSES from routes where it's installed. If login routes don't have `csrfGuard()` in the middleware chain, the cookie never gets set, and the first mutation 403s with no recourse. The "first-time login is exempt" carve-out (`isEstablished` check) means it's safe to put `csrfGuard()` on the login route itself | Add `.use(csrfGuard())` to the login route (`spike/src/app.ts:117`). Documented as landmine #6 inline | Mandatory pattern: every session-using route — including login — must include `csrfGuard()` in the chain. A `withSession()` wrapper that bundles `session(...)` + `csrfGuard()` would prevent this entirely. |
| **L6 (new)** | "Typed client" is path-typed only. `Leaf<T = unknown>`, `Input.body: unknown`. The generated `Client` interface has full path-tree autocomplete but zero per-route input/output typing. Compile-time errors on body shape mismatch will NOT fire | None — for the spike, this is acceptable. **Production fix:** since the route definition holds the Zod schemas (`spike/src/app.ts:23-27`), an in-house typed-client wrapper can extract the schema at import time and produce per-route `Leaf<RequestType, ResponseType>` signatures. This is one-evening work and would restore the "typed" claim | This is the single largest delta from the SDD/PRD claim "typed RPC client." Worth a Sprint 1.1 task to write the in-house typed wrapper before consumers (honey-road, Sietch) start hitting `as unknown as` casts. |
| **L7 (new)** | Malformed bearer JWT (token that isn't even base64-decodable as JSON) → **500** `JSON Parse error`, not the expected 401. `auth-jwt/jwt.ts:verifyJwt` throws a generic `Error`/`SyntaxError` for parse failures; the middleware only catches `JwtError` (`try { verifyJwt(...) } catch (e) { if (e instanceof JwtError) return 401 }`). Invalid signatures DO 401 correctly; only the parse-error path leaks | None in spike. Production: wrap `verifyJwt` in a try/catch that also catches `SyntaxError`/`TypeError` and returns 401. Trivial vendored-source patch | Auth surface should never 500 on attacker-controlled input. Add a regression test (`spike/test/`-style) before production; capture as a vendored-source patch the framework upstream should accept |
| **L8 (new)** | `bunx hyper openapi` CLI invocation does NOT pick up the `openapiHandlers(app, {converters:[zodConverter]})` configuration. CLI runs its own `generate(app, {})` against the app's route manifest without the registered Zod converter; output is missing all request body schemas (just `"schema":{"type":"object"}`). The RUNTIME `/openapi.json` endpoint DOES have full schemas because the app constructs `openapiHandlers` with the converter | Use the runtime endpoint (`curl /openapi.json`) for the canonical spec, NOT the CLI. CI artifact emit: scrape `/openapi.json` while the app is running, don't run `bunx hyper openapi` | Affects build pipeline: spec emit needs the app booted, not a static generator pass. Slightly awkward for CI but not broken. Alternative: write a `scripts/emit-openapi.ts` that imports `openapiHandlers` directly with the converter, mimicking the runtime path |
| **L9 (new)** | Typed client codegen elides routes that share a path-segment with a namespace. E.g., a route `GET /users` and a route `GET /users/:id` BOTH exist at runtime (`bunx hyper routes` confirms 3 routes), but the typed client emits only `api.get.users.id` (the namespace `users` overshadows the bare `users` leaf). The bare `/users` is reachable via direct fetch but not via the generated client | None in spike. Production avoidance: name collisions are predictable — for every parent-namespaced route (`GET /things`) that also has children (`GET /things/:id`), provide either an explicit `meta.name` to disambiguate, or design routes to never collide (`GET /things/list` + `GET /things/:id`) | Affects API design for identity-api. The `getIdentity(userId)` (FR-R4, `GET /v1/identity/:userId`) doesn't have this issue. The `resolve` family (`GET /v1/resolve/wallet/:address` etc.) also doesn't collide. But naming conventions should be set so we don't trip on this later. |

## The one-def claim: holds — with one important asterisk

**YES** — a single Hyper route definition produces all four artifacts:

| Artifact | Source | How derived | Quality |
|----------|--------|-------------|---------|
| **Runtime handler** | `route.METHOD(path).body(zodSchema).handle(fn)` | Direct | ✓ Full — validation, error envelope, structured logs all work |
| **OpenAPI 3.1 spec** | `openapiHandlers(app, {converters:[zodConverter]})` → mounted at `/openapi.json` | `body(zodSchema)` flows into `zodConverter.toJsonSchema(s)` → embedded in `paths.*.requestBody.content."application/json".schema` | ✓ Full for request schemas; ✗ response schemas only carry `description:"success"` (no body schema). Real architectural gap — the framework doesn't yet have a `returns(schema)` builder method, so 200 response bodies aren't declared. |
| **Typed RPC client** | `bunx hyper client <out>` (or live-watch in `hyper dev`) | Reads the route graph at build time, emits a nested-namespace tree mirroring the path structure | ✓ Path/method autocomplete works; ✗ `Input.body: unknown` and `Leaf<T = unknown>` mean NO per-route body or response typing. **The biggest delta from the SDD/PRD claim "one-def → typed client."** |
| **MCP surface** | Per-route opt-in via `meta: { mcp: { title, description } }`; `app.toMCPManifest()` reads it; `mcpServer(app)` serves JSON-RPC 2.0 over HTTP POST or stdio | ✓ Full — tool name, title, description, request/response funnel through `app.invoke()` so middleware+validation run exactly once per tool call. The MCP per-route opt-in design is correct (no accidental tool exposure) |

**Asterisk:** the "typed" half of "typed RPC client" is shallow — path-typed only, not body- or response-typed. If we want compile-time-safe consumer code (which the SDD §4.3 implies with `import { resolveUser } from '@0xhoneyjar/identity'`), we need to either (a) post-process the client codegen to attach per-route Zod types, (b) write a thin in-house typed wrapper that imports both the route file and uses TypeScript inference to derive types, or (c) wait for upstream Hyper to ship a typed-client mode. None are blockers; (b) is the lowest-friction Sprint 1.1 task.

| Need | Hyper component | Verified |
|------|-----------------|----------|

| Encrypted-cookie session | `@hyper/session` with pluggable `SessionStore` (default in-memory; `@hyper/session/sqlite` ships too) | ✓ Verified end-to-end (`spike/src/app.ts:97-101, 117-129`) |

**Integration sketch for identity-api (SDD §5.2 endpoints, ready to write):**

```ts
// src/auth.ts — central re-export that pre-installs .auth()
import { route } from "@hyper/core"
import { authJwt, installAuthMethod } from "@hyper/auth-jwt"
import { session, csrfGuard, memorySessions } from "@hyper/session"

const jwtMw = authJwt({ secret: env.JWT_SECRET, algorithms: ["ES256"] })
installAuthMethod(jwtMw)
export { route }

export const withSession = () => [
  session({ secret: env.SESSION_SECRET, cookieName: "idapi_sess", store: pgSessions() }),
  csrfGuard(),
]
```

Then handlers across the spine are:

```ts
import { route, withSession } from "./auth"
const verify = route.post("/v1/auth/verify").body(VerifyReq).use(...withSession()).handle(...)
const me = route.get("/v1/me").auth().handle(...)        // .auth() now type-visible
```

This is what Sprint 1.1 should land alongside the JWTSigner-port integration; everything else in SDD §5.2 / §5.3 / §5.5 follows the same shape.

## Open questions / risks for the operator

1. **ES256 not in the changelog.** `auth-jwt` ships `HS256/RS256` per the CHANGELOG; the SDD specifies **ES256** (D7, FR-J2). I have not confirmed ES256 support — the spike used HS256. **Action:** before Phase 1.1, either (a) verify ES256 is in `auth-jwt/jwt.ts` (likely — Web Crypto supports it), (b) PR upstream to add it, or (c) replace `@hyper/auth-jwt` with an in-house ES256 verifier using `jose` (the building's existing `adapters/jwks-validator.ts` already uses `jose`). Recommendation: (c) is lowest-risk + reuses existing code; only the `.auth()` sugar would need to be re-installed. ETA <1 day.
2. **JWKS issuance is OUR problem.** Hyper has JWT verification but no JWKS endpoint generator + key-rotation tooling. SDD FR-J2 requires "own JWKS at `/.well-known/jwks.json`, overlap-window key rotation" — this is in-house code, not a Hyper plugin. The building's existing `jwt-service.ts` lifted from loa-freeside is the starting point.
3. **The 5 typecheck errors in vendored Hyper source need a sustained plan.** Options: (a) carry a small patch set in `spike/src/hyper/` and document them in `hyper.lock.json` drift; (b) PR upstream and wait; (c) loosen `tsc` config (`skipLibCheck: true` already, plus `exclude: ["src/hyper/**"]` for the strict pass). Recommendation: (a) — they're small (3-line type-cast tweaks) and Hyper is moving fast (commits same day as the spike). Risk: every `bunx hyper update` will need a diff-review.
4. **The CLI vs runtime OpenAPI emit split is brittle for CI.** `bunx hyper openapi` produces a schema-less spec. The canonical spec is the runtime endpoint. For CI artifact emit (publishing the OpenAPI doc to the building's docs or feeding it to consumers' typed-client codegen), we need either (a) a `scripts/emit-openapi.ts` that imports `openapiHandlers` directly + the converter set + writes to disk, or (b) start the app in CI and `curl /openapi.json > openapi.json`. Recommendation: (a), trivial to write.
5. **Hyper is very young.** Public 2026-04-23, 120 stars at spike time, single maintainer (`pontusab`), still on `0.1.x`. CHANGELOG promises "API shape is expected to be stable for the 0.x line; behavior changes will be called out" — that's a soft commitment, not a contract. The source-distributed model partially insulates us (we own a snapshot), but `hyper update` may carry breaking changes. **Mitigation:** treat `hyper.lock.json` SHA-pins as immutable until an intentional `update` sprint task. Capture every patched file in a `spike/PATCHES.md`-style log.
